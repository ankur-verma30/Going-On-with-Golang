########################################
🔷 HTTP Requests with HTTP Client in Go
########################################
Go’s HTTP client is in the net/http package.

🔄 Key Types
➤ http.Client → main client
➤ http.Request → request object
➤ http.Response → response object


🔄 Simple GET Request
package main

import (
    "fmt"
    "io"
    "net/http"
)

func main() {
    resp, err := http.Get("https://api.example.com/data")
    if err != nil {
        panic(err)
    }
    defer resp.Body.Close()

    body, _ := io.ReadAll(resp.Body)
    fmt.Println(string(body))
}

⚡ Note and Tips
1️⃣ Always close response body
2️⃣ Body is an io.Reader


🔄 Using http.Client (Recommended)
✅ Why Not http.Get()?
No timeout control ❌
No customization ❌

✅ Proper Way
client := &http.Client{
    Timeout: 5 * time.Second,
}
resp, err := client.Get("https://api.example.com")
👉 Always use custom client with timeout


🔄 Creating Custom Requests
✅ POST Request
payload := strings.NewReader(`{"name":"ankur"}`)

req, _ := http.NewRequest("POST", "https://api.example.com", payload)
req.Header.Set("Content-Type", "application/json")

client := &http.Client{}
resp, err := client.Do(req)

👉  Flow: Request → Client → Server → Response


🔄 Reading Response
defer resp.Body.Close()

body, _ := io.ReadAll(resp.Body)
fmt.Println(string(body))

⚠️ Important
Always call: defer resp.Body.Close()


🔄 Handling Status Codes
if resp.StatusCode != http.StatusOK {
    fmt.Println("Request failed")
}

✔️ Common Codes
➤ 200 → OK
➤ 201 → Created
➤ 400 → Bad Request
➤ 500 → Server Error


🔄 JSON Handling with HTTP
1️⃣ Decode JSON Response
var data map[string]interface{}
json.NewDecoder(resp.Body).Decode(&data)

2️⃣ Better Approach (Struct)
type User struct {
    Name string `json:"name"`
}

var u User

🔄 Sending JSON in Request

req.Header.Set("Content-Type", "application/json")


🔄 Custom Transport
❓ What is Transport?
➤ Handles low-level network stuff
➤ Connection pooling
➤ TLS settings

✅ Example

    Transport: &http.Transport{
        MaxIdleConns:    10,
        IdleConnTimeout: 30 * time.Second,
    },
}

🔁 Why Important?
➤ Improves performance
➤ Reuses connections


🔄 Timeout Handling
❌ Bad (default client)
Can hang forever

✅ Good
client := &http.Client{
    Timeout: 10 * time.Second,
}


🔄 Context-Based Requests
✅ Example

🔄 Connection Reuse
✅ Go automatically:
    ✔️ Reuses TCP connections
    ✔️ Uses keep-alive
👉 BUT only if: resp.Body.Close()


🔄 Common Mistakes
❌ Not closing body ─────────▶ Causes memory leaks
❌ No timeout ─────────▶ Production disaster
❌ Ignoring errors
❌ Using default client everywhere ─────────▶ No control


🔄 Best Practices
➤ Use single reusable client
➤ Set timeout
➤ Use context
➤ Use struct for JSON
➤ Handle status codes properly


🔄 Interview Questions

#######################################
🔷 Building an HTTP Server with GoLang
#######################################
🔄 Core Abstractions
✅ http.Handler
type Handler interface {
    ServeHTTP(http.ResponseWriter, *http.Request)
}
👉 Any type implementing this can serve HTTP

✅ http.HandlerFunc (Adapter)
type HandlerFunc func(http.ResponseWriter, *http.Request)
Lets you use functions as handlers


🔄 Mental Model
Incoming Request → Handler → ResponseWriter → Client


🔄 Minimal Server
func handler(w http.ResponseWriter, r *http.Request) {
    w.Write([]byte("Hello World"))
}

func main() {
    http.HandleFunc("/", handler)
    http.ListenAndServe(":8080", nil)
}

❓ What Happens Internally
➤ Opens TCP listener on :8080
➤ Accepts connections
➤ Spawns goroutine per request
➤ Calls your handler


🔄 Request Handling
1️⃣ Access Key Data
r.Method        // GET, POST
r.URL.Path      // route
r.URL.Query()   // query params
r.Header        // headers
r.Body          // request body (io.Reader)

2️⃣ Read Body Safely
defer r.Body.Close()
body, err := io.ReadAll(r.Body)

⚠️ Trap
Body is a stream (io.Reader) → can be read only once


🔄 Response Handling
1️⃣ Write Response
w.Write([]byte("response"))
2️⃣ Set Headers
w.Header().Set("Content-Type", "application/json")
3️⃣ Set Status Code
w.WriteHeader(http.StatusCreated)

⚠️ Rule
✔️ WriteHeader() must be called before Write()
✔️ If not → defaults to 200 OK


🔄 JSON API
type User struct {
    Name string `json:"name"`
}

func handler(w http.ResponseWriter, r *http.Request) {
    var u User
    json.NewDecoder(r.Body).Decode(&u)

    w.Header().Set("Content-Type", "application/json")
    json.NewEncoder(w).Encode(u)
}


🔄 Routing (Default vs Advanced)
✅ Default Router

❌ Limitations
✔️ No path params (/user/{id})
✔️ No grouping

👉 Insight For production:
✔️ Use gorilla/mux (legacy)
✔️ Or frameworks like Gin / Echo


🔄 Middleware
Wrap handlers to inject behavior

✅ Example: Logging Middleware
func logging(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        fmt.Println("Request:", r.Method, r.URL.Path)
        next.ServeHTTP(w, r)
    })
}

✅ Usage
http.Handle("/", logging(http.HandlerFunc(handler)))

🔄 Real Use Cases
✔️ Authentication
✔️ Logging
✔️ Rate limiting
✔️ CORS


🔄 Custom Server
❌ Bad

✅ Good
server := &http.Server{
    Addr:         ":8080",
    Handler:      nil,
    ReadTimeout:  5 * time.Second,
    WriteTimeout: 10 * time.Second,
    IdleTimeout:  120 * time.Second,
}
server.ListenAndServe()


❓ Why Important
✔️ Prevents slow client attacks
✔️ Controls resource usage


🔄 Concurrency Model
👉 Go Automatically: Spawns goroutine per request
👉 You Must Handle:
✔️ Shared state (use sync.Mutex)
✔️ Race conditions

✅ Example Problem

🔄 Graceful Shutdown
❗Problem
Abrupt shutdown kills active requests
✅ Solution

defer cancel()
server.Shutdown(ctx)

❓ What It Does
✔️ Stops accepting new requests
✔️ Waits for existing ones to finish

###########################################
🔷 Request Routing with ServeMux (Go 1.22)
###########################################
➤ Go 1.22 upgraded net/http’s default router (ServeMux) into a powerful pattern-based router.
➤ ServeMux a route multiplexer allows you to match different request handlers to different URLs served by your servers.
➤ Used List of Registers patterns with their respective handlers.


🔄 Patterns is ServeMux
➤ Patterns are the URL patterns that ServeMux will accept.
➤ The General pattern is of form: '[METHOD][HOST]/[PATH]'
✅ Example: GET http://localhost:8080/users

❓ What Changed in Go 1.22?
👉 Before (Old ServeMux)
➤ Only exact or prefix match
➤ No path params ❌
➤ No method-based routing ❌

👉 Now (Go 1.22+)
➤ Method-aware routes
➤ Path parameters ({id})
➤ Wildcards (...)
➤ Cleaner routing → less need for frameworks


🔄 Creating a ServeMux
mux := http.NewServeMux()


🔄 Method-Based Routing
mux.HandleFunc("GET /users", getUsersHandler)
mux.HandleFunc("POST /users", createUserHandler)

👉 Insight
➤ Route includes HTTP method + path
➤ Eliminates manual if r.Method == ...


🔄 Path Parameters
1️⃣ Define Route
mux.HandleFunc("GET /users/{id}", getUserHandler)


2️⃣ Access Parameter
id := r.PathValue("id")
🔥 Example
func getUserHandler(w http.ResponseWriter, r *http.Request) {
    id := r.PathValue("id")
    fmt.Fprintf(w, "User ID: %s", id)
}


🔄 Wildcard Routing (...)
➤ The wildcard matches only a single path segment till the next forward slash '/'.
➤ If the wildcard ends with ellipses '...', it matches all remaining path segments.
➤ A '...' wildcard is only allowed at the end of the URL.

1️⃣ Match Remaining Path
mux.HandleFunc("GET /files/{path...}", fileHandler)

✅ Example
Request: /files/a/b/c.txt
path := r.PathValue("path")
// "a/b/c.txt"


🔄 Static vs Dynamic Matching Priority
👉 Priority Rules
1️⃣ Exact match
2️⃣ Parameter match ({id})
3️⃣ Wildcard (...)

⚡ Trap
mux.HandleFunc("/users/me", handler1)
mux.HandleFunc("/users/{id}", handler2)
👉 /users/me → goes to handler1 (exact match wins)


🔄 Fallback (404 Handling)
1️⃣ Default Behavior
If no route matches → 404 Not Found

2️⃣ Custom 404
mux.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
    http.NotFound(w, r)
})


🔄 Mounting ServeMux to Server
server := &http.Server{
    Addr:    ":8080",
    Handler: mux,
}
server.ListenAndServe()


🔄 Middleware with ServeMux
1️⃣ Wrap Entire Mux
loggedMux := loggingMiddleware(mux)
http.ListenAndServe(":8080", loggedMux)

2️⃣ Wrap Specific Routes
mux.Handle("GET /secure", authMiddleware(http.HandlerFunc(handler)))


🔄 Real Example (Production Style)
mux := http.NewServeMux()

mux.HandleFunc("GET /users", func(w http.ResponseWriter, r *http.Request) {
    fmt.Fprintln(w, "All users")
})

mux.HandleFunc("GET /users/{id}", func(w http.ResponseWriter, r *http.Request) {
    id := r.PathValue("id")
    fmt.Fprintf(w, "User: %s\n", id)
})

mux.HandleFunc("POST /users", func(w http.ResponseWriter, r *http.Request) {
    fmt.Fprintln(w, "User created")
})
http.ListenAndServe(":8080", mux)


🔄 Internal Working
1️⃣ Matching Algorithm
➤ Patterns are parsed into a routing tree
➤ Matching is:
    ✔️ Efficient
    ✔️ Deterministic
    ✔️ No regex overhead


❓ Why This Matters
➤ Faster than many third-party routers
➤ Minimal allocations


🔄 Limitations
❌ No advanced features like:
➤ Route groups
➤ Middleware chaining built-in
➤ Complex validation
👉 For large apps → still use:
    ✔️ Gin
    ✔️ Echo


🔄 Common Mistakes
❌ Forgetting method in route
"/users" // old style (still works but less precise)

❌ Wrong param access
r.URL.Query().Get("id") // ❌ WRONG for path params

❌ Route conflicts
Overlapping patterns without understanding priority


🔄 Best Practices
➤ Always use method-based routing
➤ Prefer path params over query for resource IDs
➤ Keep handlers small → move logic to services
➤ Use middleware for cross-cutting concerns


🔄 Trap Questions
❓ What’s new in Go 1.22 ServeMux?
Method-based routing + path params + wildcards

❓ How to access path parameters?
r.PathValue("id")

❓ Priority order in routing?
Exact > param > wildcard

❓ Why use ServeMux instead of frameworks?
Lightweight, fast, built-in

❓ When NOT to use ServeMux?
Complex routing (middlewares, grouping)


🔄 Advanced Insight
👉  Go’s Direction
✔️ Moving toward minimal built-in frameworks
✔️ Reduce dependency on external routers

👉  Real Production Pattern
✔️ ServeMux → Middleware → Handler → Service → DB


#######################################
🔷 Mastering GoLang's Context Package
#######################################
The context package is fundamental for production Go. It enables cancellation, timeouts, deadlines, and request-scoped data propagation across goroutines.

➤ Conext allows you to signal that the operation can be cancelled.
➤ You can associate a deadline with a context, specifying the maximum time for an operation to complete
➤ Context can be used to carry arbitrary data along with the function calls.

❓ Why context Exists
In concurrent systems:
➤ Requests may take too long
➤ Clients may disconnect
➤ Goroutines may leak

👉 You need a way to:
➤ Cancel work ❌
➤ Set deadlines ⏱️
➤ Pass request-scoped metadata
👉 Solution: context.Context


❓ What is context.Context?
type Context interface {
    Deadline() (deadline time.Time, ok bool)
    Done() <-chan struct{}
    Err() error
    Value(key any) any
}

🔄 General Rules
➤ Context creation should be done at the root of the execution call.
➤ By convention context is the first parameter of a function.
➤ Context should not be saved in variable for later use.
➤ Context should not be mutated.


🔄 Responsibilities
➤ Cancellation signal
➤ Timeout/deadline
➤ Carry request-scoped values


🔄 Uses
➤ HTTP Request context can carry values through middlewares and handlers.
➤ Database operations can rely on context for cancellation.
➤ Long running goroutines can be cancelled with context.


🔄 Creating Contexts
✅ Root Contexts
ctx := context.Background()
ctx := context.TODO()

🔁 Difference
➤ Background() → production root
➤ TODO() → placeholder (during development)


🔄 Cancellation
1️⃣ Create Cancelable Context
ctx, cancel := context.WithCancel(context.Background())
defer cancel()

2️⃣ Listen for Cancellation
select {
case <-ctx.Done():
    fmt.Println("Cancelled:", ctx.Err())
}

🔄 Use Case
➤ Stop goroutines when request ends


🔄 Timeout & Deadline
1️⃣ Timeout
ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
defer cancel()

2️⃣ Deadline
ctx, cancel := context.WithDeadline(context.Background(), time.Now().Add(2*time.Second))

🔥 Difference
| Type     | Meaning       |
| -------- | ------------- |
| Timeout  | Relative time |
| Deadline | Absolute time |


🔄 Context in HTTP
1️⃣ Request Context
func handler(w http.ResponseWriter, r *http.Request) {
    ctx := r.Context()
}

👉 Key Insight
➤ Automatically cancelled when:
    ✔️ Client disconnects
    ✔️ Request times out


✅ Example: Long Task
func handler(w http.ResponseWriter, r *http.Request) {
    ctx := r.Context()

    select {
    case <-time.After(5 * time.Second):
        fmt.Fprintln(w, "Done")
    case <-ctx.Done():
        fmt.Println("Request cancelled:", ctx.Err())
    }
}


🔄 Passing Context
✅ Rule
👉 Always pass context as first parameter
func process(ctx context.Context)

❌ Wrong
func process(ctx context.Context, ...)
👉 (Actually correct but must be first param; wrong is placing it later)

✅ Example
func fetchData(ctx context.Context) error {
    select {
    case <-time.After(3 * time.Second):
        return nil
    case <-ctx.Done():
        return ctx.Err()
    }
}


🔄 Context Values
✅ Store Value
ctx := context.WithValue(context.Background(), "userID", 123)
✅ Retrieve
id := ctx.Value("userID")

🔄 Best Practice
➤ Use only for:
    ✔️ Request metadata
    ✔️ Auth info
➤ NOT for:
    ✔️ Business logic data ❌
👉 Trap: Keys should be custom types, not strings


🔄 Context Tree
1️⃣ Parent → Child
parent := context.Background()
child, cancel := context.WithCancel(parent)

2️⃣ Behavior
➤ Cancelling parent → cancels child
➤ Child does NOT affect parent


🔄 Common Patterns
✅ Worker with Context
func worker(ctx context.Context) {
    for {
        select {
        case <-ctx.Done():
            fmt.Println("Stopping worker")
            return
        default:
            fmt.Println("Working...")
            time.Sleep(1 * time.Second)
        }
    }
}


🔄 Common Mistakes
❌ Not calling cancel() ─────────▶ Leads to memory leaks
❌ Passing nil context
func f(ctx context.Context) {
    // ctx == nil ❌
}
❌ Using context for optional params
❌ Ignoring ctx.Done()


🔄 Best Practices
➤ Always pass context explicitly
➤ Always call cancel()
➤ Use timeouts for external calls
➤ Never store large data in context


🔄 Internal Working
👉 Implementation
Uses:
    ✔️ Channels (Done())
    ✔️ Immutable chaining
    ✔️ Tree structure


🔄 Cancellation Mechanism
➤ Closing channel → broadcast signal
➤ All listeners stop immediately


🔄 Questions
❓ Why context is needed?
─────────▶ To manage cancellation & timeouts in concurrent systems


❓ Difference between WithCancel, WithTimeout, WithDeadline?
─────────▶ ✔️ Cancel → manual
    ✔️ Timeout → relative
    ✔️ Deadline → absolute


❓ Why pass context as first parameter?
─────────▶ Convention + readability


❓ Can context be stored in struct?
─────────▶ ❌ No (anti-pattern)


❓ What happens if you don’t call cancel()?

🔄 Table-Driven Tests
👉 Go’s most preferred testing pattern

✅ Example
func TestAdd(t *testing.T) {
    tests := []struct {
        name     string
        a, b     int
        expected int
    }{
        {"positive", 2, 3, 5},
        {"zero", 0, 5, 5},
        {"negative", -1, -1, -2},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            result := Add(tt.a, tt.b)
            if result != tt.expected {
                t.Errorf("expected %d, got %d", tt.expected, result)
            }
        })
    }
}

❓ Why Important
➤ Clean
➤ Scalable
➤ Easy to extend

🔄 Testing HTTP Handlers

import (
    "net/http"
    "net/http/httptest"
    "testing"
)

func handler(w http.ResponseWriter, r *http.Request) {
    w.Write([]byte("ok"))

func TestHandler(t *testing.T) {
    req := httptest.NewRequest("GET", "/", nil)
    rr := httptest.NewRecorder()

    handler(rr, req)

    if rr.Code != http.StatusOK {
        t.Errorf("expected 200, got %d", rr.Code)
    }
}

➤ Write small, focused tests
➤ Use interfaces for mocking
➤ Prefer table-driven tests
➤ Keep tests deterministic
➤ Use httptest for APIs
➤ Add benchmarks for critical paths


🔄 Internal Working
1️⃣ go test does:
➤ Compiles test binary
➤ Runs tests in isolation
➤ Supports parallel execution

2️⃣ Naming Convention
➤ Only functions starting with Test, Benchmark, Example are executed


🔄 Trap Questions
❓ What is table-driven testing?
─────────▶ Testing multiple cases using struct slice

❓ How to test HTTP handlers?
─────────▶ Use httptest

❓ How to mock dependencies?
─────────▶ Use interfaces

❓ Difference between Errorf and Fatalf?
─────────▶ Errorf continues, Fatalf stops

❓ How to run benchmarks?
─────────▶ go test -bench=.


🔄 Real-World Testing Strategy
👉 Pyramid: Unit Tests → Integration Tests → E2E Tests

🔁 In Go Projects
➤ Unit → functions, services
➤ Integration → DB, APIs
➤ E2E → full system
