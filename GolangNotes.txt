###########################
🔷 Introduction to Golang
###########################
Go (also called Golang) is:
    ✔️️ A statically typed
    ✔️️ compiled
    ✔️️ concurrent
    ✔️ garbage-collected
programming language.

Designed to:
    ✔️ Build fast
    ✔️ scalable
    ✔️ reliable
    ✔️ production-grade
systems with simple code


🔁 Who Created Go & Why?
👨‍💻 Creators (Google, 2007)
1️⃣ Robert Griesemer
2️⃣ Rob Pike
3️⃣ Ken Thompson (creator of Unix & C)


❓ Why Go Was Create
Google engineers faced problems with:
➤ Slow compile times in C++ ❌
➤ Complex syntax and abstractions ❌
➤ Poor concurrency support ❌
➤ Hard-to-maintain large codebases ❌

👉 Goal of Go: “A language as fast as C, as readable as Python, and as safe as Java — but simpler.”


🧠 Go Design Philosophy
| Principle                    | Meaning                           |
| ---------------------------- | --------------------------------- |
| Simplicity                   | Less syntax, fewer keywords       |
| Readability                  | Code should be readable by anyone |
| Explicitness                 | No hidden magic                   |
| Composition over inheritance | Interfaces + structs              |
| Concurrency as first-class   | Built into the language           |
| Fast builds                  | Designed for large codebases      |


🏗 Go Architecture (VERY IMPORTANT)
🔁 High-Level Go Architecture
Go Source Code (.go)
        ↓
      Compiler
        ↓
     Machine Code
        ↓
   Go Runtime (GC, Scheduler)
        ↓
     Operating System


🔁 1. Go Compiler
    ✔️ Ahead-of-Time (AOT) compiled
    ✔️ No JVM / no bytecode
    ✔️ Produces single native binary

Advantages
    ✔️ Fast startup 🚀
    ✔️ No runtime dependency
    ✔️ Easy deployment

✅ Example:
go build main.go
./main


🔁 2. Go Runtime
Unlike Java, Go runtime is small and focused.
Go Runtime Manages:
    ✔️ Goroutines scheduling
    ✔️ Garbage Collection
    ✔️ Memory allocation
    ✔️ Channel communication
    ✔️ Stack management
⚠️ You don’t manage runtime directly — Go does.


🔁 3. Go Scheduler (GMP Model)
The Famous G-M-P Model
| Component | Meaning                 |
| --------- | ----------------------- |
| G         | Goroutine               |
| M         | OS Thread               |
| P         | Processor (logical CPU) |

G (goroutine)
   ↓
P (scheduler)
   ↓
M (OS thread)

❓ Why this matters?
➤ Millions of goroutines can run on few OS threads
➤ Extremely lightweight concurrency
➤ No thread explosion


🔁 4. Memory Architecture
Stack vs Heap in Go
➤ Goroutines start with small stack (~2KB)
➤ Stack grows/shrinks dynamically
➤ Escape analysis decides:
    ✔️ Stack allocation ✅
    ✔️ Heap allocation ❌
👉 Improves performance significantly.

🌊 First Program of Golang
############################
✅ Example: First Program
package main

import "fmt"

func main(){
	fmt.Println("Hello World!")
}


🔁 package main
A package is:
    ✔️ A unit of code organization
    ✔️ A namespace
    ✔️ A compilation boundary
Every .go file must belong to exactly one package.


❓ Why package main?
➤ main is special in Go.
➤ If a program has:
    ✔️ package main
    ✔️ func main()
➡️ Go treats it as an executable program, not a library.
❌ Without package main, Go produces a library, not a runnable binary.
👉 Rule:Only one main package per executable.


🔁 import "fmt"
fmt = format
Part of Go’s standard library

Used for:
    ✔️ Printing
    ✔️ Formatting strings
    ✔️ Input/output

❓ Why Quotes "fmt"?
Go imports packages by path

"fmt" means:
    ✔️ Look inside Go’s standard library
    ✔️ Find the fmt package


🔷 Import System Design
➤ No wildcard imports
➤ No unused imports allowed ❌

import "fmt" // ✅ used
import "math" // ❌ compile-time error if unused
➡️ Forces clean code


🔁 func main()
func keyword defines a function

Syntax:
func functionName(parameters) returnType {
}

❓ Why is main() Special?
➤ Entry point of Go program
➤ Execution always starts from main()
➤ Cannot:
    ✔️ Take arguments
    ✔️ Return values
❌ This is invalid: func main() int { }
👉 Execution Flow: OS ─────────▶ Go Runtime ─────────▶ main()


🔁 Curly Braces { }
➤ Curly braces {} for blocks
➤ No indentation-based blocks like Python

⚠️ Brace Placement Rule
func main() { // ✅ same line

❌ This fails:
func main()
{ // ❌ syntax error

👉 Reason:
    ✔️ Go uses automatic semicolon insertion
    ✔️ Newline ends the statement


🔁 fmt.Println("Hello World!")
| Part             | Meaning      |
| ---------------- | ------------ |
| fmt              | Package name |
| Println          | Function     |
| ("Hello World!") | Argument     |


🔷 Println
Prints:
    ✔️ Value
    ✔️ Followed by newline (\n)
fmt.Print("Hi")    // no newline
fmt.Println("Hi") // newline


🔷 Internal Working
✔️ String literal "Hello World!"
✔️ Passed to Println
✔️ Formatted
✔️ Written to stdout
✔️ Flushed to terminal


🔁 String Literals in Go
"Hello World!"
✔️ Immutable
✔️ UTF-8 encoded
✔️ Stored in read-only memory


🔁 Semicolons (Hidden Feature)
You did not write semicolons, but Go inserts them automatically.
✅ Example:
fmt.Println("Hello")
Becomes internally: fmt.Println("Hello");


🔁 Program Execution (Step-by-Step)
🔗 Compile: go build main.go
🔗 Run:./main


🔁 Execution Flow
1️⃣ OS loads binary
2️⃣ Go runtime initializes
3️⃣ Garbage Collector setup
4️⃣ Goroutine scheduler starts
5️⃣ main() goroutine runs
6️⃣ Println executes
7️⃣ Program exits


🔁 Why No #include Like C?
Go uses:
    ✔️ Packages
    ✔️ Modules
    ✔️ Explicit imports
➡️ Faster compilation
➡️ No header files
➡️ No cyclic dependencies

###############################
🌊 Primitive Data Types in Go
###############################
➤ Built-in types provided by Go
➤ Represent single, simple values
➤ Stored by value
➤ Fast, memory-efficient


👉 Go primitive types fall into 5 major categories.
1️⃣ Integer Types
Go provides explicit-sized integers to avoid ambiguity.

🔄 Signed Integers
| Type  | Size           | Range         |
| ----- | -------------- | ------------- |
| int8  | 8-bit          | -128 to 127   |
| int16 | 16-bit         | -32768 to 32767   |
| int32 | 32-bit         | -2³¹ to 2³¹-1 |
| int64 | 64-bit         | -2⁶³ to 2⁶³-1 |
| int   | Arch dependent | 32 or 64-bit  |


🔄 Unsigned Integers
| Type   | Size           | Range        |
| ------ | -------------- | ------------ |
| uint8  | 8-bit          | 0 to 255     |
| uint16 | 16-bit         | 0 to 65K     |
| uint32 | 32-bit         | 0 to 2³²-1   |
| uint64 | 64-bit         | 0 to 2⁶⁴-1   |
| uint   | Arch dependent | 32 or 64-bit |

👉 Special Aliases
byte = uint8
rune = int32
➤ byte ─────────▶ raw data
➤ rune ─────────▶ Unicode character
✅ Example:
var myByte byte='A' // ─────────▶ byte is an alias for uint8
fmt.Println("Value of myByte is",myByte)

var myRune rune='😁' // ─────────▶ rune is an alias for int32 for unicode characters
fmt.Println("Value of myRune is",myRune)

👉 Aliases
x := 10 // type is int

🚫 Traps
➤ int size varies by architecture
➤ Mixing int and int64 ❌ (explicit cast required)
➤ Unsigned underflow can cause bugs


🔄 Type Casting
➤ TypeCasting rule types case the right side value into the left side value type
➤ Type casting can only be possible between compatible data types

➤ Syntax:
<variable_name>=<target_data_type>(<value_to_be_converted>)
✅ Example:
var a int8 = 100
var b int16 = int16(a)


2️⃣ Floating-Point Types
Used for decimal numbers.

🔄 Float Types
| Type             | Size   | Precision  |
| -------          | ------ | ---------- |
| float32          | 32-bit | ~7 digits  |
| float64(default) | 64-bit | ~15 digits |

🔄 Floating Precision Problem
0.1 + 0.2 != 0.3 // true
Reason: Binary floating-point representation

🔄 Best Practice
➤ Use float64 for most cases
➤ Avoid float for money (use integers)


3️⃣ Boolean Type
var isActive bool = true

🔄 Properties
➤ Only true or false
➤ No truthy/falsy values
➤ Cannot convert int ─────────▶ bool

❌ Invalid: if 1 {} // int is not bool


4️⃣ Character Type (Rune)
Go does not have char.

🔄 Rune
var ch rune ='A'

➤ Alias for int32
➤ Stores Unicode code point
➤ Uses single quotes

🔄 Unicode Support 
var smile rune = '😀'
Go is Unicode-first language


5️⃣ String Type
➤ Immutable sequence of bytes
➤ UTF-8 encoded
➤ Stored as:
    ✔️ Pointer
    ✔️ Length


🔄 String Declaration
s := "Hello World"

🔄 String vs Rune vs byte
s := "Go😊"
len(s) // bytes, not characters

🔄 Immutable Nature
s[0] = 'X' ─────────▶ ❌ compile error

🔄 String Literals
1️⃣ Interpreted String
s := "Hello\nWorld"

2️⃣ Raw String
s := `Hello
World`


🔄 Types of strings
➤ A one line string can be created using "". To add line breaks you would need to add "\n" escape character for new line.
➤ A multi-line string can be created using backticks ``. It preserves the formatting including line breaks and spaces.
➤ Strings can be Formatted with string formatters.


6️⃣ Complex Numbers (Rare but Built-In)
c := complex(3, 4)
| Type       | Description         |
| ---------- | ------------------- |
| complex64  | float32 real + imag |
| complex128 | float64 real + imag |

🔄 Use Case
➤ Scientific computing
➤ Signal processing


🔁 Zero Values
Go automatically initializes variables.
| Type   | Zero Value |
| ------ | ---------- |
| int    | 0          |
| float  | 0.0        |
| bool   | false      |
| string | ""         |
| rune   | 0          |

✅ Example
var x int
fmt.Println(x) // 0


🔁 Type Conversion (No Implicit Casting)
Go is strict.

❌ Invalid:
var a int = 10
var b int64 = a

✔️ Valid: var b int64 = int64(a)


🔁 Memory Representation 
✔️ Primitives stored directly
✔️ Passed by value
✔️ Copy occurs on assignment


✅ Example
a := 10
b := a
b = 20
─────────▶ a remains 10.


🔁 Best Practices
➤ Use int for general integers
➤ Use float64 for decimals
➤ Use rune for characters
➤ Avoid uint unless needed
➤ Use raw strings for regex, JSON

🔷 Variables
==============
🔄 Variable
➤ A named memory location
➤ Stores a value of a specific type
➤ Go is statically typed


🔄 Ways to Declare Variables
1️⃣ var Declaration (Explicit)
var a int
a = 10

🔗 With Initialization
var b int = 20

🔗 Type Inference(Implicit)
var c = 30 // int

🔗 Multiple Variables + Implicit
var x, y = 1, 2

🔗 Grouped Declaration
Used to group similar type of variables
var (
	name string = "Go"
	age  int    = 15
)


2️⃣ Short Variable Declaration :=
➤ The keyword var can be omitted along with the type which is inferred from the value on right in variable declaration.
➤ Can be used to declare multiple variables as long as one variable on the left is new.
d := 40

👉 Rules
➤ Only inside functions
➤ At least one new variable must be declared
➤ Compiler infers type

❌ Invalid: x := 10 // at package level

🔗 Mixed Assignment (Trap ⚠️)
a := 10
a, b := 20, 30 // ❌ a is not new variable
✔️ Correct: a, b := 10, 20


🔄 Variable Scope
1️⃣ Local Variables
➤ Declared inside functions
➤ Accessible only within block

2️⃣ Package-Level Variables
var appName = "GoApp"

➤ Shared across files in same package
➤ Initialized before main()


3️⃣ Block Scope
if x := 10; x > 5 {
	fmt.Println(x)
}
👉 x exists only inside if block


🔄 Shadowing (Common Bug)
x := 10
if true {
	x := 20 // shadows outer x
}
👉 Outer x remains 10.


====================
🔷 Constants in Go
====================
🔄 Constant
➤ Value known at compile time
➤ Immutable
➤ No memory allocation at runtime and cannot make a runtime value constant
➤ In Go constants can only hold values that the compiler can infer to be constant at the compile time.


🔄 Constant values
1️⃣ Boolean values i.e. true or false
2️⃣ Numeric values for example 1, 2, 3,110.01, etc.
3️⃣ Global functions like complex, real, imag, len and cap
4️⃣ Strings and rune values for example "Hello",'a' etc.
5️⃣ Expressions that can be evaluated at compile time for example 1<2.

🔗 Declaring Constants
const pi = 3.14

🔗 Typed Constant ─────────▶ can only be assigned to directly to a variable of the same kind
const age int = 25

🔗 Untyped Constant (Powerful) ─────────▶ Type can be inferred or it fallbacks to default type for the constant value.
const x = 10

🔗 Why Untyped Constants Matter
const x = 10
var y int64 = x // allowed

🔗 Multiple Constants
const (
	A = 1
	B = 2
	C = 3
)

🔗 iota – Constant Generator 🔥
const (
	A = iota // 0
	B        // 1
	C        // 2
)


✅ Practical Example
const (
	_ = iota
	Monday
	Tuesday
	Wednesday
)

✅ iota with Expressions
const (
	KB = 1 << (10 * iota)
	MB
	GB
)


🔄 Constant Rules (Strict)
❌ Invalid: const x = getValue() // function call
✔️ Valid: const x = 10 + 20


🔄 What Types Can Be Constants?
✅ Allowed:
➤ Numbers
➤ Strings
➤ Booleans

🚫 Not allowed:
➤ Slices
➤ Maps
➤ Structs


🔄 Similarities between variables and constants
1️⃣ package-level and function-level
2️⃣ Can be grouped using braces ().
3️⃣ Can be initialized using :=


🔄 Differences between variables and constants
| Feature       | Variable  | Constant  |
| ------------- | --------- | --------- |
| Mutability    | Mutable   | Immutable |
| Compile-time  | ❌        | ✔️       |
| Memory        | Allocated | None      |
| Function call | ✔️       | ❌        |
| iota          | ❌       | ✔️        |


🔄 Initialization Order
1️⃣ Package-level variables
2️⃣ init() function
3️⃣ main()
func init() {
	fmt.Println("Init runs first")
}


🔄 Best Practices
➤ Use := for local variables
➤ Use var for package-level variables
➤ Use const wherever possible
➤ Avoid shadowing
➤ Group related declarations

🔄 Keywords
➤ Keywords are reserved words
➤ They have special meaning in the Go language
➤ You cannot use them as variable names, function names, package names, etc.

🔄 List of Go Keywords (25)
Go has only 25 keywords (small on purpose).

1️⃣ Control Flow
if, else, switch, case, default, for, break, continue, goto, fallthrough, return

2️⃣ Declarations / Definitions
var, const, type, func, package, import

3️⃣ Concurrency
go, select, chan, defer

4️⃣ Structuring / Composition
struct, interface, map

5️⃣ Special
range
👉 Notice: Go intentionally has few keywords, so code stays simple and readable.


🔄 Keywords vs Predeclared Identifiers
Many people confuse these.

1️⃣ Keywords (Reserved)
✅ Example: func, var, for

2️⃣ Predeclared Identifiers (NOT keywords)
These are built-in names that you can override (but you shouldn’t):

✅ Examples:
int, string, bool
true, false
nil
make, new, len, cap, append, copy, close
print, println


✅ Example (allowed but bad practice):
func main() {
	len := 100
	fmt.Println(len) // prints 100
}
🚫 This shadows built-in len() function ─────────▶ confusing.


================
🔷 Identifiers
================
An identifier is the name you give to: variables, constants, functions, types (structs, interfaces), packages, fields/methods, labels

✅ Examples:
var age int
const Pi = 3.14
func calculateSum() {}
type User struct {}
👉 Here: age, Pi, calculateSum, User are identifiers


🔄 Rules for Valid Identifiers
Rule 1️⃣: Must start with a letter or _
Valid: name, _age, User1
Invalid ❌: 1user, @name


Rule 2️⃣: Can contain letters, digits, underscore
Valid:
✔️ user_name_1
✔️ totalSum2

✅ Rule 3️⃣: Case-sensitive
var name = "A"
var Name = "B"
➤ name and Name are different identifiers.


✅ Rule 4️⃣: Cannot be a keyword
❌ Invalid: var for = 10


4️⃣ Exported vs Unexported Identifiers
Go uses a simple rule:
🔥 Capital Letter = Exported (Public)

Accessible from other packages.
type User struct {
	Name string // exported field
}

🔒 Small Letter = Unexported (Private)
Accessible only within same package.

type user struct {
	name string // unexported field
}


✅ Example (Package Boundary)
mathutils/sum.go
package mathutils

func Add(a, b int) int { // exported
	return a + b
}

func subtract(a, b int) int { // unexported
	return a - b
}


🔗 main.go
package main
import (
	"fmt"
	"myapp/mathutils"
)

func main() {
	fmt.Println(mathutils.Add(2, 3)) // ✅ works
	// mathutils.subtract(5, 2)      // ❌ not accessible
}

5️⃣ Naming Conventions (Go Style)
Go has strong naming culture.

✅ Variables
1️⃣ short but meaningful
2️⃣use camelCase
userName := "Ankur"
totalMarks := 95

✅ Functions
func calculateTotal() {}
func fetchUserByID() {}

✅ Types (Struct / Interface)
PascalCase (exported)
nouns

type UserService struct {}
type PaymentGateway interface {}


✅ Interfaces (Go Convention)
Many interfaces end with er

type Reader interface {
	Read([]byte) (int, error)
}
type Writer interface {
	Write([]byte) (int, error)
}


❌ Avoid these
snake_case (not idiomatic)
Hungarian notation
overly long names

6️⃣ Blank Identifier _ (Special Identifier)
➤ Go has a special identifier: _
➤ Meaning: "I don’t care about this value”
➤ Used to ignore values or avoid unused-variable errors
✅ Example: x, _ := fmt.Println("Hello")


7️⃣ Common Questions / Traps
✅ Q1: Is int a keyword?
❌ No. int is a predeclared identifier, not a keyword.


✅ Q2: Can we name a variable nil?
❌ No, nil is a predeclared identifier and should not be shadowed.


✅ Q3: What decides public/private in Go?
➡️ First letter case
Name ─────────▶ public/exported
name ─────────▶ private/unexported


✅ Q4: What is _ used for?
➡️ Ignore values + avoid unused warnings.

🔷 Operators in Golang
########################
➤ A symbol that performs an operation on one or more operands
➤ Go operators are simple, explicit, and predictable
➤ No operator overloading (except + for strings)


1️⃣ Arithmetic Operators
Used for mathematical calculations.
| Operator | Meaning        |
| -------- | -------------- |
| `+`      | Addition       |
| `-`      | Subtraction    |
| `*`      | Multiplication |
| `/`      | Division       |
| `%`      | Modulus        |

✅ Example
a := 10
b := 3

fmt.Println(a + b) // 13
fmt.Println(a - b) // 7
fmt.Println(a * b) // 30
fmt.Println(a / b) // 3 (integer division)
fmt.Println(a % b) // 1

⚠️ Interview Traps
1️⃣ Integer Division
fmt.Println(10 / 4) // 2, NOT 2.5
Fix: fmt.Println(float64(10) / float64(4))

2️⃣ Modulus Rules
% works only with integers
❌ 10.5 % 2 // invalid


2️⃣ Relational (Comparison) Operators
Used for comparisons ─────────▶ result is bool.
| Operator | Meaning          |
| -------- | ---------------- |
| `==`     | Equal            |
| `!=`     | Not equal        |
| `>`      | Greater          |
| `<`      | Less             |
| `>=`     | Greater or equal |
| `<=`     | Less or equal    |

✅ Example
fmt.Println(5 > 3)   // true
fmt.Println(5 == 3)  // false

⚠️ Interview Traps
➤ Structs can be compared only if all fields are comparable
➤ Slices, maps, functions ❌ cannot be compared

a := []int{1,2}
b := []int{1,2}
fmt.Println(a == b) // ❌ compile error


3️⃣ Logical Operators
Used with boolean expressions.
| Operator | Meaning |
| -------- | ------- |
| `&&`     | AND     |
| `||      | OR      |
| `!`      | NOT     |
✅ Example
age := 20
fmt.Println(age > 18 && age < 30) // true


📌 Short-Circuit Evaluation
false && panic("won’t run")
true || panic("won’t run")
👉 Very important for nil checks.


4️⃣ Assignment Operators
| Operator | Meaning           |
| -------- | ----------------- |
| `=`      | Assign            |
| `+=`     | Add & assign      |
| `-=`     | Subtract & assign |
| `*=`     | Multiply & assign |
| `/=`     | Divide & assign   |
| `%=`     | Modulus & assign  |

✅ Example
x := 10
x += 5
fmt.Println(x) // 15


❌ No ++x or --x in Go
Only postfix is allowed:
x++ // valid
++x // invalid
👉 And ++ / -- are statements, not expressions.


5️⃣ Bitwise Operators
Operate at bit level.
| Operator | Meaning     |
| -------- | ----------- |
| `&`      | AND         |
| `|`      | OR          |
| `^`      | XOR         |
| `<<`     | Left shift  |
| `>>`     | Right shift |
| `&^`     | AND NOT     |

✅ Example
a := 5  // 0101
b := 3  // 0011

fmt.Println(a & b)  // 1
fmt.Println(a | b)  // 7
fmt.Println(a ^ b)  // 6

🔥 AND NOT (&^) – Go Specific
a := 10 // 1010
b := 3  // 0011
➤ fmt.Println(a &^ b) // 8 (1010 & 1100)
➤ Used in flags & masks.


6️⃣ Unary Operators
Operate on single operand.
| Operator | Meaning     |
| -------- | ----------- |
| `+`      | Unary plus  |
| `-`      | Unary minus |
| `!`      | Logical NOT |
| `^`      | Bitwise NOT |
| `*`      | Dereference |
| `&`      | Address-of  |

✅ Example
x := 10
fmt.Println(-x) // -10


7️⃣ Pointer Operators
x := 10
p := &x   // address-of
fmt.Println(*p) // dereference

➤ No pointer arithmetic in Go ❌
➤ Safer than C/C++


8️⃣ Type Operators
1️⃣ Type Conversion
var a int = 10
var b float64 = float64(a)

2️⃣ Type Assertion (Interface)

Safe version: s, ok := i.(string)


9️⃣ Channel Operators (Concurrency)

x := <-ch  // receive
👉 Used only with chan.


🔟 Operator Precedence(PEMDAS -> Parantheses, Exponents, Multiplication, Division and Substraction)
Highest ─────────▶ Lowest (Simplified)
* / % << >> & &^
+ - | ^
== != < <= > >=
&&
||

✅ Example

1️⃣1️⃣ Operators NOT Present in Go
❌ No:

    ✔️ Pre-increment ++x

🔷 Arrays in Golang
#####################
An array is:
    ✔️ A fixed-size
    ✔️ contiguous
    ✔️ homogeneous collection of elements
Size is part of the type ❗

var arr [5]int


🔄 Declaring Arrays
1️⃣ Basic Declaration
var arr [5]int

➤ Creates array of 5 integers
➤ Zero-initialized ─────────▶ [0 0 0 0 0]


2️⃣ Declaration with Initialization
arr := [5]int{1, 2, 3, 4, 5}


3️⃣ Partial Initialization
arr := [5]int{1, 2}
Result: [1 2 0 0 0]


4️⃣ Length Inference (...)
arr := [...]int{10, 20, 30}
Compiler determines size ─────────▶ [3]int


5️⃣ Index-Based Initialization
arr := [5]int{0: 10, 3: 40}
Result: [10 0 0 40 0]


🔄 Accessing & Modifying Elements
arr := [3]int{10, 20, 30}
fmt.Println(arr[0]) // 10
arr[1] = 50


⚠️ Bounds Check (Runtime Panic)
fmt.Println(arr[3]) // panic
➤ Go performs runtime bounds checking ─────────▶ safety.


🔄 Length of Array
len(arr)

➤ Returns fixed size
➤ Compile-time known


🔄 Arrays are Value Types
1️⃣ Array Assignment Copies Entire Array
a := [3]int{1, 2, 3}
b := a

b[0] = 100
fmt.Println(a) // [1 2 3]


2️⃣ Passing Arrays to Functions
func modify(arr [3]int) {
	arr[0] = 100
}

func main() {
	a := [3]int{1,2,3}
	modify(a)
	fmt.Println(a) // unchanged
}
Reason: Full copy passed

✅ Solution (Pointer)
func modify(arr *[3]int) {
	arr[0] = 100
}


🔄 Comparing Arrays
✔️ Allowed if:
    ➤ Same length
    ➤ Same type
    ➤ Elements are comparable

a := [3]int{1,2,3}
b := [3]int{1,2,3}
fmt.Println(a == b) // true

❌ Invalid: [3]int != [4]int


🔄 Iterating Over Arrays
1️⃣ Using for
for i := 0; i < len(arr); i++ {
	fmt.Println(arr[i])
}


2️⃣ Using range (Preferred)
for index, value := range arr {
	fmt.Println(index, value)
}

Ignore Index or Value
for _, v := range arr {}
for i := range arr {}


🔄 Multidimensional Arrays
1️⃣ 2D Array
var matrix [2][3]int

2️⃣ Initialization
matrix := [2][2]int{
	{1, 2},
	{3, 4},
}

3️⃣ Access
fmt.Println(matrix[1][0]) // 3


🔄 Memory Layout
➤ Stored in contiguous memory
➤ Cache-friendly
➤ Size known at compile time

Stack vs Heap
Small arrays ─────────▶ stack
Large arrays ─────────▶ heap (escape analysis)


🔄 When to Use Arrays in Go?
✅ Use arrays when:
    ✔️ Size is fixed
    ✔️ Performance critical
    ✔️ Low-level programming
    ✔️ Embedded / system-level work

❌ Avoid arrays for:
    ✔️ Dynamic collections
    ✔️ API boundaries


🔄 Arrays vs Slices
| Feature              | Array | Slice          |
| -------------------- | ----- | -------------- |
| Size                 | Fixed | Dynamic        |
| Type includes length | ✔     | ❌            |
| Pass by value        | ✔     | Reference-like |
| Commonly used        | ❌    | ✔             |
| Flexible             | ❌    | ✔             |


🔄 Common Interview Traps
❓ Why arrays are rarely used?
➡️ Fixed size + copy semantics.


❓ Is array passed by reference?
❌ No, passed by value.


❓ Can arrays be resized?
❌ No.


❓ Why slices exist?
➡️ To solve array limitations.


##########################
🔷 Unveiling Slices in Go
##########################
➤ A dynamic view over an array
➤ A descriptor, not a data container itself
➤ The most commonly used collection type in Go

✅ Declaration
var arr = []int{1, 2, 3}


🔄 Slice vs Array
| Feature              | Array | Slice             |
| -------------------- | ----- | ----------------- |
| Size                 | Fixed | Dynamic           |
| Type includes length | ✔     | ❌               |
| Passed by value      | ✔     | Descriptor copied |
| Real-world usage     | Rare  | Very common       |


🔄 Slice Internal Structure
A slice internally has three components:
type slice struct {
    pointer  *T   // points to array
    length   int
    capacity int
}
⚠️ Key Insight: Passing a slice copies this struct, NOT the underlying array.


🔄 Visual Representation
Underlying Array: [10 20 30 40 50]
Slice:             ↑
pointer = index 1
len = 2
cap = 4


🔄 Declaring & Initializing Slices
1️⃣ Nil Slice
var s []int

➤ len(s) == 0
➤ cap(s) == 0
➤ s == nil ─────────▶ true


2️⃣ Empty Slice
s := []int{}

➤ len == 0
➤ cap == 0
➤ s != nil


3️⃣ Using make
s := make([]int, 5)

➤ Length = 5
➤ Capacity = 5

s := make([]int, 3, 10)

➤ Length = 3
➤ Capacity = 10


4️⃣ Slice Literal
s := []int{1, 2, 3}


🔄 Slicing an Array or Slice
arr := [5]int{10, 20, 30, 40, 50}
s := arr[1:4]

Result: [20 30 40]


1️⃣ Slice Expression Rules
s := arr[low : high]

➤ Includes low
➤ Excludes high


Full Slice Expression (Advanced)
s := arr[1:3:4]

➤ len = high - low
➤ cap = max - low


🔄 Length vs Capacity
s := []int{1,2,3,4}
fmt.Println(len(s)) // 4
fmt.Println(cap(s)) // 4

Capacity Growth (append)
s := []int{}
s = append(s, 1)

➤ Capacity grows automatically
➤ Growth strategy is implementation-dependent


🔄 append() – How It Really Works
Case 1️⃣: Capacity Available
➤ Element added
➤ Same underlying array


Case 2️⃣: Capacity Full
➤ New array allocated
➤ Old data copied
➤ Slice now points to new array


✅ Trap Example
a := []int{1,2,3}
b := append(a, 4)

a[0] = 100
fmt.Println(b) // ?
⚠️ Depends on capacity!


🔄 Slices are Reference-Like
func modify(s []int) {
    s[0] = 100
}

func main() {
    a := []int{1,2,3}
    modify(a)
    fmt.Println(a) // [100 2 3]
}
❓ Why?
➤ Slice descriptor copied
➤ Underlying array shared


🔄 Copying Slices Correctly
src := []int{1,2,3}
dst := make([]int, len(src))
copy(dst, src)

❌ Wrong Copy
dst := src // shares same array


🔄 Iterating Over Slices
for i, v := range s {
    fmt.Println(i, v)
}

⚠️ Common Trap
for _, v := range s {
    v = 100 // does NOT modify slice
}

✅ Correct:
for i := range s {
    s[i] = 100
}


🔄 Deleting Element from Slice
i := 2
s = append(s[:i], s[i+1:]...)


Slice vs nil vs empty
| Property | nil slice | empty slice |
| -------- | --------- | ----------- |
| len      | 0         | 0           |
| cap      | 0         | 0           |
| == nil   | ✔         | ❌         |
| JSON     | null      | []          |


🔄 Performance & Memory Insights
➤ Slices avoid copying large data
➤ Appending repeatedly may cause reallocations
➤ Pre-allocate when size known
s := make([]int, 0, 1000)


🔄 Common Interview Questions
❓ Why slices are not comparable?
➡️ They contain pointers.


❓ Why slices are passed by reference?
➡️ Underlying array shared.


❓ Difference between len and cap?
➡️ Length = visible elements, Capacity = max before realloc.


##############################
🔷 “Ripple Effect” in Golang
##############################
It means that: Changes made to one slice may unintentionally affect another slice, because both slices share the same underlying array.

This happens due to Go's slice architecutre.


🔄 Before Ripple Effect ─────────▶ Understand Slice Internals (CRITICAL)
➤ A slice is NOT an array.
➤ Internally, a slice is a descriptor (header) with three fields:

type slice struct {
    pointer  *array  // points to underlying array
    length   int
    capacity int
}
So when you write: s := []int{10, 20, 30, 40}

Memory looks like: Underlying Array: [10, 20, 30, 40]
                   ↑
                   s (len=4, cap=4)


🔄 Creating a Slice From Another Slice
s1 := []int{10, 20, 30, 40}
s2 := s1[1:3]

Slice Header Values
| Slice | Points To | Len | Cap |
| ----- | --------- | --- | --- |
| s1    | index 0   | 4   | 4   |
| s2    | index 1   | 2   | 3   |

Memory:
Underlying Array: [10, 20, 30, 40]
                    ↑        ↑
                   s2       s2 end
👉 Same array, different view


✅ Ripple Effect Example
package main

import "fmt"

func main() {
    s1 := []int{10, 20, 30, 40}
    s2 := s1[1:3]

    s2[0] = 999

    fmt.Println("s1:", s1)
    fmt.Println("s2:", s2)
}
✅ Output
s1: [10 999 30 40]
s2: [999 30]


❓ Why did s1 change?
➤ s2[0] points to s1[1]
➤ Same underlying array
➤ Modification ripples back
➡️ This is the Ripple Effect


🔄 Visual Explanation
s1 = [10, 20, 30, 40]
          ↑
        s2[0]
👉 Changing s2[0] modifies index 1 of the array ─────────▶ reflected in s1.


🔄 Ripple Effect with append()
Case 1️⃣: Capacity Available ─────────▶ Ripple Happens
s1 := []int{1, 2, 3, 4}
s2 := s1[1:3] // len=2, cap=3

s2 = append(s2, 999)

fmt.Println(s1)
fmt.Println(s2)

✅ Output
[1 2 3 999]
[2 3 999]

❓ Why?
➤ s2 had extra capacity
➤ append reused same underlying array
➤ 999 placed at index 3
➤ s1 affected


Case 2️⃣: Capacity Exhausted ─────────▶ NO Ripple
s1 := []int{1, 2, 3, 4}
s2 := s1[1:3:3] // Full slice expression ─────────▶ cap limited

s2 = append(s2, 999)

fmt.Println(s1)
fmt.Println(s2)

✅ Output
[1 2 3 4]
[2 3 999]

❓ Why?
➤ Capacity restricted
➤ New array allocated
➤ No shared memory
➤ No ripple


🔄 Full Slice Expression
s2 := s1[low : high : max]
➤ len = high - low
➤ cap = max - low
➤ Prevents accidental overwrite


🔄 Avoiding Ripple Effect
Method 1️⃣: Use copy()
original := []int{10, 20, 30}
safe := make([]int, len(original))
copy(safe, original)

Now:
    ✔️ Separate arrays
    ✔️ No ripple


Method 2️⃣: Force new array with append
safe := append([]int{}, original...)

❌ Dangerous Pattern (COMMON BUG)
a := []int{1, 2, 3}
b := a[:2]
b[0] = 100
👉 Developers expect a unchanged — WRONG

❓ Why Go Allows This
Go prioritizes:
    ✔️ Performance
    ✔️ Low allocations
    ✔️ Predictable memory usage

Sharing arrays:
    ✔️ Avoids copying
    ✔️ Faster slicing
    ✔️ Responsibility on developer
👉 Go gives you power, not safety by default


❓ Questions
1️⃣ Why does modifying one slice affect another?
✔️ Because slices share the same underlying array.


2️⃣ When does append allocate a new array?
✔️ When capacity is insufficient.


3️⃣ How to avoid ripple effect?
✔️ Use copy() or restrict capacity using full slice expression.


4️⃣ Is slice pass-by-value or reference?
✔️ Slice header is passed by value, but it points to shared memory.


################################################
🔷 Connections between Strings, Runes and Bytes
################################################
In Go, strings, bytes, and runes are tightly connected, but they are NOT the same thing.
| Concept  | Represents                  | Size              |
| -------- | --------------------------- | ----------------- |
| `string` | Read-only sequence of bytes | variable          |
| `byte`   | Raw binary data             | 1 byte (`uint8`)  |
| `rune`   | A Unicode code point        | 4 bytes (`int32`) |
Go strings are UTF-8 encoded byte sequences


🔄 String in Go?
s := "Hello 🌍"

Internally: A string is:
type string struct {
    ptr *byte
    len int
}
➤ Immutable
➤ Stored as UTF-8 encoded bytes


🔗 Length of a String
fmt.Println(len("Hello"))     // 5
fmt.Println(len("🌍"))        // 4 ❗

❓ Why?
➤ 🌍 is 1 character
➤ But 4 bytes in UTF-8


🔄 Byte: The Lowest Level
b := []byte("Hello 🌍")
fmt.Println(b)

Output: [72 101 108 108 111 32 240 159 140 141]

👉 Key Points
➤ Each number is a byte
➤ UTF-8 characters may take 1–4 bytes
➤ byte = alias for uint8


Iterating Over Bytes (❌ for Unicode)
for i := 0; i < len(s); i++ {
    fmt.Printf("%c ", s[i])
}
Output: H e l l o   � � � �
🚨 Broken for Unicode


🔄 Rune: Unicode Code Point
r := '🌍'
fmt.Println(r)        // 127757
fmt.Printf("%T", r)   // int32

👉 Key Points
➤ rune = alias for int32
➤ Represents one Unicode character
➤ Can hold any Unicode symbol

Iterating Over Runes (✅ Correct)
for _, ch := range "Hello 🌍" {
    fmt.Printf("%c ", ch)
}
Output: H e l l o   🌍


🔄 range — The Connector Between Them
When you use range on a string:
for i, r := range s {
    fmt.Println(i, r)
}

❓ What happens internally?
➤ Decodes UTF-8 bytes
➤ Converts to rune
➤ Moves index by rune byte length
➡️ range = UTF-8 aware


🔄 String ─────────▶ Rune Slice ─────────▶ String
s := "Go🌍"
r := []rune(s)
fmt.Println(len(s))   // 6 bytes
fmt.Println(len(r))   // 3 runes

❓ Why convert?
➤ Safe character indexing
➤ Replacing characters

✅ Example: Modify a String (Strings are immutable)
s := "Hello 🌍"
r := []rune(s)
r[6] = '🚀'
s = string(r)
fmt.Println(s)

✅ Output: Hello 🚀


🔄 String Indexing Pitfall
s := "🌍"
fmt.Println(s[0])

✅ Output: 240

❓ Why?
─────────▶ s[0] gives first byte,Not the rune

✅ Correct Way
r := []rune(s)
fmt.Println(r[0])     // 127757
fmt.Printf("%c", r[0])


🔄 Visual Memory Model
"🌍" (string)

Use string when:
➤ Read-only text
➤ Logging

Use []byte when:

➤ Network data
➤ Performance-critical code

Use []rune when:

################################
🔷 Demystifying Maps in Golang
################################
Maps are one of the most used data structures in Go — especially in backend, APIs, caching, configuration, and data processing.

If you master maps, you master:
    ✔️ Hashing behavior
    ✔️ Reference semantics
    ✔️ Concurrency pitfalls
    ✔️ Performance optimization


🔄 What is a Map in Go?
A map is a built-in data structure that associates keys with values. It is implemented as a hash table and provides efficient lookups, insertions, and deletions.
key ─────────▶ value

✅ Example:
m := map[string]int{
    "apple":  10,
    "banana": 20,
}

🔄 Declaring and Initializing Maps
1️⃣ Zero Value Map
var m map[string]int

➤ m == nil ─────────▶ true
➤ Cannot insert values into nil map ─────────▶ panic
➤ Reading from nil map returns zero value ─────────▶ 0
➤ Writing to nil map causes runtime panic
m["key"] = 100 // ❌ panic: assignment to entry in nil map


🔄 Initialization of Empty Map
1️⃣ Using Literal
m := map[string]int{}
➤ Now it is initialized and can be used safely.

2️⃣ Using make()
m := make(map[string]int)
m["apple"] = 10
➤ Now it is initialized and can be used safely.

3️⃣var m map[string]int= map[string]int{}


🔄 With Initial Capacity
m := make(map[string]int, 100)
➤ Pre-allocates space for 100 entries
➤ Performance boost if size known
➤ Capacity is a hint, not a fixed size


4️⃣ Map Literal
m := map[string]int{
    "apple":  10,
    "banana": 20,
}


🔄 Map Internal Architecture
Go maps are implemented as hash tables with separate chaining for collision resolution.
Internally: key ─────────▶ hash function ─────────▶ bucket ─────────▶ value

🔁 Core Concepts
➤ Uses hashing
➤ Data stored in buckets
➤ Each bucket stores multiple key-value pairs
➤ Buckets linked when overflow happens
➤ Automatically resizes (rehashes)


🔄 Key Rules for Map Keys
1️⃣ Must be Comparable types
✅ Allowed: bool, int, float, string, pointers, structs (if all fields comparable)
❌ Not allowed: slices, maps, functions


🔄 Reading From Map
v:= m["key"]
➤ If key exists ─────────▶ returns value
➤ If key does not exist ─────────▶ returns zero value of value type (e.g., 0 for int, "" for string)


🔄 Comma-OK Idiom
v, ok := m["key"]
➤ ok == true ─────────▶ key exists
➤ ok == false ─────────▶ key does not exist
This is crucial to distinguish between a missing key and a key with zero value.

✅ Example:
m := map[string]int{
    "apple": 10,
}
v, ok := m["banana"]
fmt.Println(v)  // 0 (zero value)
fmt.Println(ok) // false (key does not exist)


🔄 Writing to Map
m["key"] = value
➤ If key exists ─────────▶ updates value
➤ If key does not exist ─────────▶ creates new entry


🔄 Deleting from Map
delete(m, "key")
➤ Removes key-value pair if key exists
➤ No error if key does not exist


🔄 Iterating Over Map
for k, v := range m {
    fmt.Println(k, v)
}
➤ Iteration order is random ─────────▶ do NOT rely on it
➤ Each run may produce different order


🔄 Map Are Reference Types
When you assign a map to another variable, both variables point to the same underlying data structure.
m1 := map[string]int{
    "apple": 10,
}
m2 := m1
m2["banana"] = 20
fmt.Println(m1) // map[apple:10 banana:20]
fmt.Println(m2) // map[apple:10 banana:20]

Map behaves like: slices, pointers, channels, functions (reference semantics).


🔄 Passing Maps to Functions
func modify(m map[string]int) {
    m["x"] = 500
}
Change is visible outside the function because map header is passed by value but underlying structure is a reference type.


🔄 Comparing Maps
Maps cannot be compared directly using == operator (except for nil).
❌ m1 == m2 // compile error
✔️ Only comparison allowed is with nil: m == nil

✅ To compare maps, you must write a custom function that checks keys and values.


🔄 Concurrency Warning
Maps are NOT safe for concurrent use by default. If multiple goroutines access a map simultaneously and at least one of them modifies the map, it can lead to a race condition and cause a runtime panic.
fatal error: concurrent map writes

If multiple goroutines access map:
    ✔️ Use sync.Mutex
    ✔️ Or sync.RWMutex
    ✔️ Or sync.Map


🔄 When to use sync.Map
✅ Use sync.Map when:
    ✔️ High contention
    ✔️ Many goroutines
    ✔️ Read-heavy workloads
    ✔️ When you want built-in concurrency safety without manual locking


🔄 Map Resizing and Growth
When:
    ✔️ Load factor increases
    ✔️ Too many collisions

Go:
    ✔️ Allocates new larger bucket array
    ✔️ Gradually rehashes elements
    ✔️ Uses incremental growth
👉 This prevents: Big latency spikes


🔄 Zero Value Behavior
✅ Example:
m := make(map[string]int)
fmt.Println(m["unknown"])

✅ Output: 0
👉 Because: Zero value of int is 0

For bool map:
m := make(map[string]bool)
fmt.Println(m["x"]) // false


🔄 Map of Structs vs Map of Pointers
map[string]User
map[string]*User

Use pointer version when:
    ✔️ Struct is large
    ✔️ Frequent modifications
    ✔️ Performance sensitive


🔄 Performance Insights
Average complexity:
    ✔️ Lookup ─────────▶ O(1)
    ✔️ Insert ─────────▶ O(1)
    ✔️ Delete ─────────▶ O(1)
Worst case:  O(n) (heavy collisions)


🟢 Questions
❓ Q1  Why can’t slice be a map key?
─────────▶ Because slices are not comparable.

❓ Q2  Why is map iteration unordered?
─────────▶ To avoid order dependency.

❓ Q3  Is map thread-safe?
─────────▶ No.

❓ Q4  Difference between nil map and empty map?
| Feature   | nil map | empty map |
| --------- | ------- | --------- |
| Can read  | ✔       | ✔        |
| Can write | ❌ panic| ✔        |
| len       | 0       | 0         |


🔄 Memory Model Summary
Map variable contains:
    ✔️ Pointer to runtime hash table
    ✔️ Count
    ✔️ Flags
👉 Real data lives in heap.


###############################
🔷 Understanding Structs in Go
###############################
➤ Structs are one of the most important data structures in Go.
➤ They allow you to group related data into a single type, similar to objects in OOP languages.
➤ But Go structs are simpler and more powerful because they work with:
    ✔️ composition
    ✔️ interfaces
    ✔️ memory efficiency
    ✔️ method receivers


❓ What Are Structs
➤ A struct is a composite data type that groups variables under one name.
➤ Think of it as a custom data structure.

✅ Example
type User struct {
	Name  string
	Age   int
	Email string
}
👉 Now User becomes a new type.


❓ Why Structs Are Used
Structs help when you want to represent real-world entities.
| Entity  | Fields                  |
| ------- | ----------------------- |
| User    | name, email, age        |
| Product | id, price, stock        |
| Order   | orderId, userId, amount |

✅ Example:
type Product struct {
	ID    int
	Name  string
	Price float64
}


🔄 Creating Struct Variables
Method 1️⃣ — Zero Value Initialization
var u User

Default values:
| Type   | Default |
| ------ | ------- |
| int    | 0       |
| string | ""      |
| bool   | false   |

✅ Example:
fmt.Println(u)
Output: { 0 }


Method 2️⃣ — Struct Literal
u := User{
	Name:  "Ankur",
	Age:   22,
	Email: "ankur@email.com",
}
👉 Preferred in production code.


Method 3️⃣ — Without Field Names (Not Recommended)
u := User{"Ankur", 22, "ankur@email.com"}

❗ Problem: Order must match struct definition.


🔄 Accessing Struct Fields
➤ Using dot operator

fmt.Println(u.Name)
fmt.Println(u.Email)

Updating value: u.Age = 23


🔄 Struct Memory Layout
Struct fields are stored sequentially in memory.

✅ Example:
type Example struct {
	A int
	B int
	C int
}
👉 Memory layout: [A][B][C]


🔄 Anonymous Structs
Sometimes you need a struct only once.

✅ Example:
user := struct {
	Name string
	Age  int
}{
	Name: "Ankur",
	Age:  22,
}

Useful for:
    ✔️ JSON parsing
    ✔️ temporary objects
    ✔️ tests


🔄 Struct Comparison
Structs are comparable if all fields are comparable.

✅ Example:
type Point struct {
	X int
	Y int
}
p1 := Point{1,2}
p2 := Point{1,2}
fmt.Println(p1 == p2) // true

❗ But if struct contains:
    ✔️ slice
    ✔️ map
    ✔️ function
👉 Comparison not allowed.


🔄 Struct Tags (Important for Backend)
Struct tags are metadata attached to fields.

✅ Example:
type User struct {
	Name  string `json:"name"`
	Email string `json:"email"`
}
Used by:
    ✔️ JSON encoder
    ✔️ ORM
    ✔️ validation libraries

✅ Example JSON:
{
 "name":"Ankur",
 "email":"ankur@email.com"
}


🔄 Embedded Structs
Go uses composition instead of inheritance.

✅ Example:
type Address struct {
	City  string
	State string
}

type User struct {
	Name string
	Address
}
👉 Now: u.City works directly.


🔄 Struct with Methods
Structs can have methods.

✅ Example:
type Rectangle struct {
	Width  int
	Height int
}

func (r Rectangle) Area() int {
	return r.Width * r.Height
}

🔁 Usage:
rect := Rectangle{10,5}
fmt.Println(rect.Area())


🔄 Pointer vs Value Struct
1️⃣ Value Receiver
func (u User) UpdateName(name string) {
	u.Name = name
}
👉 Does NOT modify original struct.

2️⃣ Pointer Receiver
func (u *User) UpdateName(name string) {
	u.Name = name
}
👉 Modifies original struct.
👉 Used in real production code.


🔄 Structs and Pointers
Create pointer to struct:
u := &User{
	Name: "Ankur",
}
➤ Access still uses dot: u.Name
➤ Go automatically dereferences.


🔄 Nested Structs
Struct inside struct.

type Address struct {
	City string
}

type User struct {
	Name string
	Addr Address
}
🔁 Usage: u.Addr.City


1️⃣4️⃣ Struct vs Class (Java Comparison)
| Feature       | Go Struct | Java Class  |
| ------------- | --------- | ----------  |
| Inheritance   | ❌        | ✔          |
| Methods       | ✔         | ✔          |
| Encapsulation | ✔         | ✔          |
| Constructors  | ❌        | ✔          |
| Composition   | ✔         | ✔          |
👉 Go prefers composition over inheritance.


🔄 Struct Best Practices (Production)
1️⃣ Use pointers for large structs
func process(u *User)
❌ Avoid copying big structs.


2️⃣ Group related data
❌ Bad:
name string
email string
age int

✅ Good: User struct


3️⃣ Use struct tags for APIs
✅ Example: json:"user_name"


🔄 Questions on Structs
❓ Are structs passed by value?
─────────▶ Yes. Entire struct copy happens.

❓ Why use pointer receivers?
─────────▶ To modify original struct and avoid copying.

❓ Can structs have methods?
─────────▶ Yes.

❓ Does Go support inheritance?
─────────▶ No — uses composition.

❓ Are structs comparable?
─────────▶ Yes, if all fields are comparable.


##############################
🔷 Go Conditional Statements
##############################
➤ Conditional statements control decision making in programs.
➤ They allow a program to execute different blocks of code depending on conditions.

➤ In Go, the main conditional statement is: if
➤ Go does not require parentheses like C/C++/Java.


🔄 Basic if Statement
🔗 Syntax
if condition {
    // code executed if condition is true
}

✅ Example
package main

import "fmt"

func main() {

    age := 20

    if age >= 18 {
        fmt.Println("Eligible to vote")
    }
}

✅ Output
Eligible to vote

🔁 Key Points
➤ condition must return boolean
➤ No parentheses required
➤ Curly braces {} are mandatory


🔄 Go Does NOT Allow Non-Boolean Conditions
➤ In languages like C: if(10)
➤ But in Go: if 10 { } // ❌ Compile error
➤ Correct: if true { }
➤ Go forces strict boolean evaluation.


3️⃣ if-else Statement
Used when you want two alternative paths.

🔗 Syntax
if condition {
    // executes if condition true
} else {
    // executes if condition false
}

✅ Example
package main

import "fmt"

func main() {

    num := 7

    if num%2 == 0 {
        fmt.Println("Even")
    } else {
        fmt.Println("Odd")
    }
}

✅ Output
Odd


🔄 if-else-if Ladder
Used when there are multiple conditions.

🔗 Syntax
if condition1 {

} else if condition2 {

} else if condition3 {

} else {

}


✅ Example
package main

import "fmt"

func main() {

    marks := 85

    if marks >= 90 {
        fmt.Println("Grade A")
    } else if marks >= 75 {
        fmt.Println("Grade B")
    } else if marks >= 50 {
        fmt.Println("Grade C")
    } else {
        fmt.Println("Fail")
    }
}

✅ Output
Grade B


🔄 Short Statement in if
Go allows variable declaration inside if.

🔗 Syntax
if variable := expression; condition {
}

✅ Example
package main

import "fmt"

func main() {

    if num := 10; num%2 == 0 {
        fmt.Println("Even:", num)
    }
}

✅ Output
Even: 10

⚡Important Rule: The variable num is scoped only inside the if block.

✅ Example:
if num := 10; num > 5 {
    fmt.Println(num)
}

fmt.Println(num) // ❌ error


🔄 Using Function Calls in if
✅ Example:
func isAdult(age int) bool {
    return age >= 18
}

func main() {

    if isAdult(21) {
        fmt.Println("Adult")
    }
}


🔄 Nested if Statements
An if inside another if.

✅ Example:

package main

import "fmt"

func main() {

    age := 25
    citizen := true

    if age >= 18 {

        if citizen {
            fmt.Println("Eligible to vote")
        }

    }
}

✅ Output
Eligible to vote


🔄 Logical Operators in Conditions
Often used inside if.
| Operator | Meaning |   |    |
| -------- | ------- | - | -- |
| `&&`     | AND     |   |    |
| `        |         | ` | OR |
| `!`      | NOT     |   |    |

✅ Example:
age := 25
salary := 60000

if age > 21 && salary > 50000 {
    fmt.Println("Loan approved")
}


🔄 Comparing Values
Comparison operators used in conditions.
| Operator | Meaning       |
| -------- | ------------- |
| `==`     | equal         |
| `!=`     | not equal     |
| `>`      | greater       |
| `<`      | less          |
| `>=`     | greater equal |
| `<=`     | less equal    |

✅ Example:
x := 10
y := 20

if x < y {
    fmt.Println("x is smaller")
}


🔄 if Without else
Perfectly valid.

if temperature > 30 {
    fmt.Println("Hot weather")
}


🔄 Common Beginner Mistakes
❌ Missing braces
Invalid:
if x > 10
    fmt.Println("hello")

✅ Correct:
if x > 10 {
    fmt.Println("hello")
}


❌ Using assignment instead of comparison
if x = 10 { } // ❌

✅ Correct:
if x == 10 { }


🔄 Backend Use case
✅ Example: API authorization.

func main() {

    role := "admin"

    if role == "admin" {
        fmt.Println("Full access")
    } else if role == "user" {
        fmt.Println("Limited access")
    } else {
        fmt.Println("Guest access")
    }
}


🔄 Performance Considerations
✅ Best practices:
    ✔️ Put most likely condition first
    ✔️ Avoid deep nesting
    ✔️ Use switch for many conditions

✅ Example:
❌ Bad: 10 nested if statements
✔️ Better: switch statement


🔄 Common Trap Questions
❓ Does Go require parentheses in if?
─────────▶ No. if x > 10 {}

❓ Can we declare variables inside if?
─────────▶ Yes. if x := 10; x > 5 {}

❓ What is the scope of variables declared in if?
─────────▶ Only inside the if block.

❓ Can condition be non-boolean?
─────────▶ No. Go requires boolean expression.


#####################################################
🔷 Understanding Code Blocks and Variable Shadowing
#####################################################
When writing Go programs, code blocks and variable shadowing are critical concepts that affect:
    ✔️ variable scope
    ✔️ memory visibility
    ✔️ unexpected bugs
    ✔️ readability of code
👉 Many production bugs in Go happen due to variable shadowing, so understanding this deeply is very important.


🔄 Code Block in Go
➤ A code block is a section of code enclosed inside: { }
➤ A block defines the scope of variables.

✅ Example
package main

import "fmt"

func main() {

    x := 10

    if x > 5 {
        y := 20
        fmt.Println(x, y)
    }

    fmt.Println(x)
}

✅ Output:
10 20
10
👉 But: fmt.Println(y) // ❌ error

❓ Why?
➤ Because y exists only inside the block.


🔄 Types of Code Blocks in Go
Blocks are created in many places.

1️⃣ Function block
func main() {

}

2️⃣ If block
if condition {

}

3️⃣ Loop block
for i := 0; i < 10; i++ {

}

4️⃣ Switch block
switch x {

}

5️⃣ Explicit block
You can even create manual blocks.
{
    x := 10
}


🔄 Variable Scope in Go
Scope means where a variable is accessible.
3 main scopes
| Scope          | Example               |
| -------------- | --------------------- |
| Package scope  | global variables      |
| Function scope | variables in function |
| Block scope    | variables inside `{}` |

✅ Example
var a = 10   // package scope

func main() {

    b := 20   // function scope

    if true {
        c := 30   // block scope
        fmt.Println(a, b, c)
    }

}


🔄 Variable Shadowing
➤ Variable shadowing happens when a variable declared in an inner scope has the same name as a variable in an outer scope.
➤ The inner variable hides (shadows) the outer one.

✅ Example
package main

import "fmt"

func main() {

    x := 10

    if true {
        x := 20
        fmt.Println("Inside:", x)
    }

    fmt.Println("Outside:", x)
}

✅ Output
Inside: 20
Outside: 10

✅ Explanation:
Outer x = 10
Inner x = 20
👉 They are different variables.


🔄 Visual Representation
main()
 ├── x = 10
 │
 └── if block
      └── x = 20 (shadows outer x)


🔄 Shadowing With Short Variable Declaration :=
Shadowing often happens with: :=

✅ Example:
x := 5

if true {
    x := x + 5
    fmt.Println(x)
}

✅ Output
10
But outer x remains: 5


🔄 Real Bug Example
func getUser() (string, error) {
    return "Ankur", nil
}

func main() {

    user := "Guest"

    if user, err := getUser(); err == nil {
        fmt.Println(user)
    }

    fmt.Println(user)
}

✅ Output:
Ankur
Guest

❓ Why?
Because the user inside if shadows the outer variable.


🔄 Shadowing With Error Variables
✅ Example:
file, err := os.Open("data.txt")

if err != nil {
    log.Fatal(err)
}

data, err := io.ReadAll(file)

👉 This is fine.
❗ But inside blocks:
if data, err := io.ReadAll(file); err != nil {
    fmt.Println(err)
}
👉 Now err is shadowed.


🔄 Preventing Shadowing Bugs
Use assignment instead of :=
❌ Wrong: x := 10

if condition {
    x := 20
}
✅ Correct: x := 10

if condition {
    x = 20
}


🔄 Go Compiler Shadowing Behavior
➤ Go allows shadowing intentionally.
➤ Unlike some languages, it is not an error.
➤ But tools like: go vet can warn about it.


🔄 Shadowing With Loops
✅ Example:
x := 100

for i := 0; i < 3; i++ {
    x := i
    fmt.Println(x)
}
fmt.Println(x)

✅ Output:
0
1
2
100


🔄 Shadowing vs Reassignment
➤ Important difference: Shadowing
➤ Creates new variable.

x := 10
{
    x := 20
}

🔁 Reassignment
Updates existing variable.
x := 10

{
    x = 20
}


🔄 Best Practices
1️⃣ Avoid reusing variable names
Bad:
x
x
x
x

2️⃣ Keep scopes small
Short blocks reduce confusion.

3️⃣ Use clear variable names
✅ Example:
userName
dbUser
apiUser


🔄 Common Questions
❓ What is variable shadowing?
─────────▶ When a variable in inner scope hides a variable with same name in outer scope.

❓ Does Go allow shadowing?
─────────▶ Yes.

❓ What operator usually causes shadowing?
─────────▶ :=

❓ How to avoid shadowing?
─────────▶ Use: = instead of :=


#####################################
🔷 Multi-Functional For Loop in Go
#####################################
➤ In Go, for is the only looping construct. Unlike many languages (Java, C++, Python), Go does not have while or do-while loops.
➤ Instead, Go's for loop is multi-functional and can behave like:
    ✔️ a traditional for loop
    ✔️ a while loop
    ✔️ an infinite loop
    ✔️ a range-based loop


🔄 Basic for Loop
🔗 Syntax
for initialization; condition; post {
    // loop body
}

✅ Example
package main
import "fmt"

func main() {

    for i := 0; i < 5; i++ {
        fmt.Println(i)
    }

}
✅ Output
0
1
2
3
4
Explanation
| Part     | Meaning         |
| -------- | --------------- |
| `i := 0` | initialization  |
| `i < 5`  | condition check |
| `i++`    | increment       |
👉 Execution order: initialization → condition → body → post → condition → body → ...


🔄 Using for as a while Loop
Go replaces while with a for loop without initialization and post.

🔗 Syntax
for condition {
}

✅ Example
package main
import "fmt"

func main() {

    i := 0

    for i < 5 {
        fmt.Println(i)
        i++
    }

}
✅ Output
0
1
2
3
4
👉 This behaves exactly like: while(i < 5)


🔄 Infinite Loop
If you remove the condition, the loop runs forever.

🔗 Syntax
for {
}

✅ Example
package main
import "fmt"

func main() {

    for {
        fmt.Println("Running forever")
    }

}

This loop only stops if:
    ✔️ break
    ✔️ return
    ✔️ panic


🔄 for with range
➤ The most commonly used loop in Go for iterating collections.
➤ Works with:
    ✔️ slices
    ✔️ arrays
    ✔️ maps
    ✔️ strings
    ✔️ channels

✅ Example with slice
nums := []int{10, 20, 30}

for index, value := range nums {

#####################################
🔷 Continue, Break, Labels, and Goto
#####################################
➤ Go provides several control flow statements that allow you to alter the normal execution of loops and blocks.
➤ The most important ones are:
    ✔️ break
    ✔️ continue
    ✔️ labels
    ✔️ goto
➤ These are used to control loops, nested loops, and program flow.


1️⃣ break Statement
➤ The break statement is used to terminate a loop or switch statement immediately.
➤ Once break executes, control moves outside the loop.

✅ Example
package main
import "fmt"

func main() {

    for i := 1; i <= 10; i++ {

        if i == 5 {
            break
        }

        fmt.Println(i)
    }
}
✅ Output
1
2
3
4

🔁 Explanation
Loop runs → when i == 5 → break → exit loop


2️⃣ continue Statement
The continue statement skips the current iteration and moves to the next iteration of the loop.

✅ Example
package main
import "fmt"

func main() {

    for i := 1; i <= 5; i++ {

        if i == 3 {
            continue
        }

        fmt.Println(i)
    }
}
✅ Output
1
2
4
5

🔁 Explanation
i == 3 → continue → skip printing


3️⃣ Labels in Go
A label is a name attached to a statement.

🔗 Syntax: LabelName:
Labels are used with:
    ✔️ break
    ✔️ continue
    ✔️ goto
They are especially useful in nested loops.


4️⃣ Labeled break
➤ Used to break out of outer loops.
➤ Without labels, break exits only the inner loop.

✅ Example
package main
import "fmt"

func main() {

OuterLoop:

    for i := 1; i <= 3; i++ {

        for j := 1; j <= 3; j++ {

            if i == 2 && j == 2 {
                break OuterLoop
            }

            fmt.Println(i, j)
        }
    }

    fmt.Println("Loop ended")
}
✅ Output
1 1
1 2
1 3
2 1
Loop ended

👉 Explanation: break OuterLoop exits both loops


5️⃣ Labeled continue
Used to continue outer loop instead of inner loop.

✅ Example
package main
import "fmt"

func main() {

OuterLoop:

    for i := 1; i <= 3; i++ {

        for j := 1; j <= 3; j++ {

            if j == 2 {
                continue OuterLoop
            }

            fmt.Println(i, j)
        }
    }
}
✅ Output
1 1
2 1
3 1
👉 Explanation: j == 2 → skip remaining inner loop → continue outer loop


6️⃣ goto Statement
The goto statement transfers control to a labeled statement.

🔗 Syntax:
goto LabelName

✅ Example
package main
import "fmt"

func main() {

    i := 1

Start:

    fmt.Println(i)

    i++

    if i <= 5 {
        goto Start
    }
}
✅ Output
1
2
3
4
5


❓ When Should You Use goto?
Go allows goto, but it should be used carefully.

Recommended uses:
    ✔️ Breaking out of deeply nested loops
    ✔️ Error handling in low-level code
    ✔️ Resource cleanup


✅ Example: Error Handling with goto
Example used in system-level code:
func process() error {

    err := step1()
    if err != nil {
        goto ERROR
    }

    err = step2()
    if err != nil {
        goto ERROR
    }

    return nil

ERROR:
    fmt.Println("Error occurred")
    return err
}


🔄 Restrictions of goto
goto cannot jump over variable declarations.

✅ Example (invalid):
goto label
x := 10
label:

Compiler error.


👉 Visual Flow Example
for loop
   |
   |--- continue → next iteration
   |
   |--- break → exit loop
   |
   |--- goto → jump to label


🔄 Best Practices
✔️ Use break for exiting loops
✔️ Use continue for skipping iterations
✔️ Use labels for nested loops
❌ Avoid excessive goto
👉 Too much goto leads to spaghetti code.


🔄 Common Trap Questions
❓ Difference between break and continue?
| break      | continue        |
| ---------- | --------------- |
| exits loop | skips iteration |

❓ Can break exit outer loop?
─────────▶ Yes, using labels.

❓ Does Go support goto?
─────────▶ Yes.

❓ Can goto jump anywhere?
─────────▶ No, it cannot jump over variable declarations.

❓ Is break needed in switch?
─────────▶ No, Go switches automatically break.

🔷 Switch Statement
####################
➤ Go’s switch is more flexible, cleaner, and safer than Java/C++ switch.
➤ It supports:
    ✔️ no break needed
    ✔️ multiple conditions
    ✔️ expression-less switch
    ✔️ type switching (VERY IMPORTANT)
    ✔️ fallthrough (optional)


🔄 Basic Switch Statement
🔗 Syntax
switch expression {
case value1:
    // code
case value2:
    // code
default:
    // code
}

✅ Example
package main
import "fmt"

func main() {

    day := 2

    switch day {
    case 1:
        fmt.Println("Monday")
    case 2:
        fmt.Println("Tuesday")
    default:
        fmt.Println("Other day")
    }
}
✅ Output
Tuesday

🔄 No break Required
👉 Unlike Java/C++:
switch x {
case 1:
    fmt.Println("One")
case 2:
    fmt.Println("Two")
}
✔️ Automatically breaks after matching case
✔️ No fall-through by default


🔄 Multiple Values in Case
num := 3
switch num {
case 1, 2, 3:
    fmt.Println("Between 1 and 3")
case 4, 5:
    fmt.Println("Between 4 and 5")
}


🔄 Expression-less Switch
You can omit the expression.

marks := 85
switch {
case marks >= 90:
    fmt.Println("Grade A")
case marks >= 75:
    fmt.Println("Grade B")
case marks >= 50:
    fmt.Println("Grade C")
default:
    fmt.Println("Fail")
}
👉 Works like if-else-if but cleaner.


🔄 Short Statement in Switch
Like if, you can declare variables inside switch.

switch num := 10; {
case num%2 == 0:
    fmt.Println("Even")
default:
    fmt.Println("Odd")
}


🔄 fallthrough Keyword
By default, Go does NOT fall through.
To force it:

x := 1
switch x {
case 1:
    fmt.Println("One")
    fallthrough
case 2:
    fmt.Println("Two")
}
✅ Output
One
Two

👉 Rules
➤ fallthrough ignores condition of next case
➤ It just executes next block blindly


🔄 Type Switch
Used with interfaces.

🔗 Syntax
switch v := x.(type) {
case int:
case string:
case bool:
default:
}
Example
package main
import "fmt"

func checkType(x interface{}) {

    switch v := x.(type) {
    case int:
        fmt.Println("Integer:", v)
    case string:
        fmt.Println("String:", v)
    default:
        fmt.Println("Unknown type")
    }
}

func main() {
    checkType(10)
    checkType("Hello")
}


🔄 Switch with Conditions
age := 25
switch {
case age < 18:
    fmt.Println("Minor")
case age >= 18 && age < 60:
    fmt.Println("Adult")
default:
    fmt.Println("Senior")
}


🔄 Switch vs If-Else
| Feature             | switch | if-else |
| ------------------- | ------ | ------- |
| Multiple conditions | ✔      | ✔      |
| Cleaner syntax      | ✔      | ❌     |
| Range checks        | ✔      | ✔      |
| Type checking       | ✔      | ❌     |


🔄 Nested Switch
x := 2
y := 3
switch x {
case 2:
    switch y {
    case 3:
        fmt.Println("x=2 and y=3")
    }
}


🔄 Empty Switch (Always True)
switch {
case true:
    fmt.Println("Always runs")
}


✅ Backend Example
role := "admin"
switch role {
case "admin":
    fmt.Println("Full access")
case "user":
    fmt.Println("Limited access")
default:
    fmt.Println("Guest")
}


🔄 Performance Insight
➤ Faster than long if-else chains
➤ Compiler may optimize into jump tables


🔄 Common Trap Questions
❓ Does Go switch need break?
─────────▶ ❌ No — automatic

❓ What is fallthrough?
─────────▶ Forces execution of next case.

❓ What is type switch?
─────────▶ Used to check dynamic type of interface

❓ Can switch work without expression?
─────────▶ ✔ Yes

❓ Difference between switch and if?
─────────▶ Switch is cleaner for multiple conditions.

🔄 Defer Keyword
➤ defer delays execution until function ends.
➤ Three conditions for deferred functions execution:
    ✔️ Surrounding function returns
    ✔️ Code execution reaches to the end of function body.
    ✔️ Function panics
➤ The deferred call's arguments are evaluated immediately, but the function call is not executed right away.
➤ Multiple deferred functions are exectued in last in first out manner(LIFO).
➤ Deferred functions are executed after any result parameter are set by that return statement but before the function returns to its caller.
➤ Used for cleanup and handling panics and resource management.

✅ Example
func main() {
    defer fmt.Println("World")
    fmt.Println("Hello")
}

✅ Output
Hello
World


🔄 Multiple Defer (LIFO Order)
defer fmt.Println("1")
defer fmt.Println("2")
defer fmt.Println("3")

Output:
3
2
1

🔗 Real Use Case
file, _ := os.Open("file.txt")
defer file.Close()

######################
🔷 Pointers in Golang
######################
➤ A pointer is a variable that stores the memory address of another variable
➤ Instead of holding a value, it points to where the value is stored

👉 Think:
➤ Normal variable → stores value
➤ Pointer → stores address of value


❓ Why Go Uses Pointers
Go is designed for performance + simplicity, so pointers help in:
1️⃣ Avoid copying large data
➤ Passing large structs → expensive (copy)
➤ Passing pointer → cheap (only address)

2️⃣ Modify original data
Functions can change actual values (not copies)

3️⃣ Efficient memory usage
Less duplication → better performance


🔗 Basic Syntax
🔄 Declaring a Pointer
var p *int
➤ *int → pointer to an integer
➤ p stores an address, not a value

🔄 Getting Address (&)
x := 10
p := &x
&x → "address of x"

🔄 Dereferencing (*)
fmt.Println(*p)
*p → value at that address


✅ Basic Example
package main

import "fmt"

func main() {
    x := 10
    p := &x

    fmt.Println("Value of x:", x)
    fmt.Println("Address of x:", p)
    fmt.Println("Value using pointer:", *p)
}


🔄 Changing Value Using Pointer
package main

import "fmt"

func main() {
    x := 10
    p := &x

    *p = 20

    fmt.Println(x) // 20
}

💡 Key Insight:
➤ *p = 20 changes the original variable
➤ No copy involved


🔄 Pass by Value vs Pass by Pointer
❌ Pass by Value (Default in Go)
func update(x int) {
    x = 50
}

func main() {
    a := 10
    update(a)
    fmt.Println(a) // 10 (unchanged)
}


✅ Pass by Pointer
func update(x *int) {
    *x = 50
}

func main() {
    a := 10
    update(&a)
    fmt.Println(a) // 50
}


🔗 Pointers with Structs
❌ Without Pointer (Copy happens)
type User struct {
    name string
}

func update(u User) {
    u.name = "Ankur"
}


✅ With Pointer (Real-world usage)
type User struct {
    name string
}

func update(u *User) {
    u.name = "Ankur"
}

func main() {
    user := User{name: "John"}
    update(&user)
    fmt.Println(user.name) // Ankur
}


⚡ Shortcut (Go Feature)
u.name = "Ankur"
Even though u is *User, Go automatically dereferences it
👉 No need to write: (*u).name = "Ankur"


🔄 Memory Behavior
👉 Stack vs Heap
➤ Small variables → Stack
➤ Large or escaping variables → Heap

👉 When you use pointers:
    ✔️ Data may escape to heap
    ✔️ Managed by Go's Garbage Collector


🔄 Escape Analysis
Go compiler decides: Should variable stay in stack Or move to heap?


🔗 Nil Pointer
var p *int
fmt.Println(p) // nil
❌ Dereferencing nil → Crash
fmt.Println(*p) // panic

✅ Safe Check
if p != nil {
    fmt.Println(*p)
}


🔄 Common Mistakes
❌ 1. Dereferencing nil pointer
→ causes runtime panic

❌ 2. Forgetting &
update(a) // wrong
update(&a) // correct

❌ 3. Overusing pointers
Makes code complex
Not always needed


🔄 When to Use Pointers
✅ Use pointers when:
    ✔️ You want to modify original data
    ✔️ Struct is large
    ✔️ Avoid copying (performance)
    ✔️ Working with shared state

❌ Avoid pointers when:
    ✔️ Data is small (int, bool)
    ✔️ You don’t need mutation
    ✔️ Simplicity matters

| Feature            | Go            | C/C++     |
| ------------------ | ------------- | --------- |
| Pointer arithmetic |❌ Not allowed| ✅ Allowed|

| Garbage Collection | Yes           | No        |
| Complexity         | Simple        | Complex   |
👉 Go intentionally restricts pointers to keep things safe

    ✔️ Caching systems

##############################################################
🔷 Call By Value | Unraveling the Mystery of Maps and Slices
##############################################################
➤ Go is 100% Call by Value — NO EXCEPTIONS
➤ Every variable is copied when passed to a function
➤ There is no pass-by-reference in Go


🔗 Then Why Do Maps & Slices Behave Like Reference?
Maps and slices internally contain pointers
So when you pass them:
    ✔️ You copy the structure
    ✔️ But that structure contains a pointer to shared data
👉 Result: Changes reflect outside the function


🔄 Internal Structure
1️⃣ Slice Internals
A slice is NOT just an array. It is:
type slice struct {
    ptr *T   // pointer to underlying array
    len int
    cap int
}
👉 When you pass a slice:
    ✔️ This struct is copied
    ✔️ But ptr still points to SAME array


2️⃣ Map Internals
A map is:
type hmap struct {
    buckets *bucket
    ...
}
👉 When you pass a map:
    ✔️ Copy of map header
    ✔️ But same underlying data (buckets)


✅ Slice Example — Looks Like Reference
package main

import "fmt"

func update(s []int) {
    s[0] = 100
}

func main() {
    arr := []int{1, 2, 3}
    update(arr)

    fmt.Println(arr) // [100 2 3]
}

❓ Why it changed?
➤ Slice copy → same underlying array
➤ s[0] modifies shared memory

🔄 Slice Edge Case
func update(s []int) {
    s = append(s, 100)
}
👉 This may NOT affect original slice

❓ Why?
➤ If capacity exceeded → new array created
➤ Now pointer changes → no longer shared

✅ Correct Way (Return Slice)
func update(s []int) []int {
    s = append(s, 100)
    return s
}


✅ Map Example — Always Feels Like Reference
package main

import "fmt"

func update(m map[string]int) {
    m["age"] = 25
}

func main() {
    m := map[string]int{"age": 20}
    update(m)

    fmt.Println(m) // map[age:25]
}

❓ Why always works?
➤ Map internally always points to shared structure
➤ No "capacity reallocation" issue like slices


🔄 Map Edge Case
func update(m map[string]int) {
    m = make(map[string]int)
    m["age"] = 50
}
👉 Output: map[age:20]
➤ You reassigned local copy
➤ Original map untouched


🔄 Slice vs Map Behavior
| Operation          | Slice            | Map              |
| ------------------ | ---------------- | ---------------- |
| Modify element     | ✅ Reflects      | ✅ Reflects     |
| Reassign variable  | ❌ No effect     | ❌ No effect    |
| Append / Grow      | ⚠️ Maybe reflect | ✅ Always reflect |
| Underlying storage | Array            | Hash buckets     |


🔄 Deep Mental Model
Everything is passed by value
But:
    ✔️ Slice → value contains pointer
    ✔️ Map → value contains pointer
👉 So: Pointer copied → still points to same memory


🔄 Common Mistakes
❌ Mistake 1: Thinking Go has pass-by-reference
👉 ❌ WRONG

❌ Mistake 2: Expecting append to always reflect
👉 ❌ WRONG (depends on capacity)

❌ Mistake 3: Reassigning map/slice inside function
👉 ❌ Does NOT affect original


✅ Real Backend Use Case
Example: Updating request data
func process(users []User) {
    for i := range users {
        users[i].Active = true
    }
}
👉 Efficient:
    ✔️ No copying of large structs
    ✔️ Direct mutation


🔄 Memory + Performance Insight
➤ Slice copy → 24 bytes (ptr + len + cap)
➤ Map copy → small header
➤ Underlying data → NOT copied
👉 This is why Go is:
    ✔️ Fast
    ✔️ Memory efficient

🔷 Types and Methods
######################
➤ A type defines the structure and behavior of data
➤ It tells:
    ✔️ What data is stored
    ✔️ What operations are allowed


🔄 Categories of Types
1️⃣ Basic Types: int, float64, string, bool
2️⃣ Composite Types: array, slice, map, struct
3️⃣ User-Defined Types: type UserID int
👉 This creates a new distinct type, not just an alias


🔄 Type vs Type Alias
type A = int   // alias
type B int     // new type

| Feature            | Alias (`A`) | New Type (`B`) |
| ------------------ | ----------- | -------------- |
| Same as original   | ✅          | ❌            |
| Can define methods | ❌          | ✅            |


🔄 Struct Types (Go’s Replacement for Classes)
type User struct {
    Name string
    Age  int
}
➤ Struct = collection of fields
➤ No inheritance
➤ Used as core building block in backend systems


🔄 Methods in Go
A method is a function attached to a type using a receiver

🔗 Syntax
func (receiver Type) methodName() returnType {
    // logic
}

✅ Example
type User struct {
    Name string
}

func (u User) greet() string {
    return "Hello " + u.Name
}


🔄 Receiver Types
1️⃣ Value Receiver
func (u User) updateName() {
    u.Name = "Ankur"
}

❌ Problem:
    ✔️ Works on a copy
    ✔️ Original object unchanged


2️⃣ Pointer Receiver
func (u *User) updateName() {
    u.Name = "Ankur"
}

🔁 Benefits:
    ✔️ Modifies original object
    ✔️ Avoids copying large structs


✅ Example
func main() {
    u := User{Name: "John"}
    u.updateName()
    fmt.Println(u.Name) // Ankur
}
👉 Go automatically converts: u.updateName() → (&u).updateName()


❓ When to Use Pointer vs Value Receiver
1️⃣ Pointer Receiver
➤ Need to modify data
➤ Struct is large
➤ Performance critical
➤ Consistency across methods

2️⃣ Value Receiver
➤ Data is small (int, bool, small structs)
➤ Immutable logic
➤ Read-only operations


🔄 Method Sets
This determines which methods a type can call

🔁 Rules
| Type | Can Call                  |
| ---- | ------------------------- |
| `T`  | methods with receiver `T` |
| `*T` | methods with `T` and `*T` |

✅ Example
type User struct{}

func (u User) m1() {}
func (u *User) m2() {}

👉 Behavior
var u User
u.m1() // ✅
u.m2() // ✅ (auto address)

var p *User
p.m1() // ✅
p.m2() // ✅


🔄 Methods on Custom Types
type MyInt int

func (m MyInt) square() int {
    return int(m * m)
}

🔗 Usage
var x MyInt = 5
fmt.Println(x.square()) // 25


🔄 Restrictions
❌ Cannot define methods on: Built-in types directly
func (i int) test() {} // ❌

✅ Solution
type MyInt int


🔄 Methods vs Functions
| Feature       | Method | Function |
| ------------- | ------ | -------- |
| Has receiver  | ✅    | ❌       |
| Bound to type | ✅    | ❌       |
| OOP behavior  | ✅    | ❌       |


🔄 Internal Working
Go compiler converts: u.greet()
👉 Into: greet(u)
👉 Methods are just syntactic sugar over functions


🔄 Real Backend Example

    Balance float64
}

func (a *Account) deposit(amount float64) {
    a.Balance += amount

Some methods pointer, some value → confusion

It won’t

─────────▶  Partially (no classes, uses structs + methods)

─────────▶  Modify original + better performance

Go avoids:
    ✔️ Inheritance

| ------------------ | ----- | ------------ |

❓ What is iota?
iota is a special constant generator used inside const blocks

✅ Basic Example
const (
    A = iota // 0
    B        // 1
    C        // 2
)

⚡ Key Rule
👉 iota:
    ✔️ Starts at 0
    ✔️ Increments by 1 per line


✅ Real Example with Derived Type + iota

    Pending Status = iota
    Approved
    Rejected
)

🔗 Usage

    ✔️ Status codes
    ✔️ User roles
    ✔️ Order states
    ✔️ HTTP states (custom)


🔗 Skipping Values
const (
    A = iota // 0
    _
    C        // 2
)


🔗 Custom Starting Value
const (
    A = iota + 1 // 1
    B            // 2
    C            // 3
)


🔄 Bitmasking
const (
    Read = 1 << iota  // 1
    Write             // 2
    Execute           // 4
)


👉 Explanation
| Name    | Value |
| ------- | ----- |
| Read    | 1     |
| Write   | 2     |
| Execute | 4     |
👉 Used in:
    ✔️ Permissions
    ✔️ Flags
    ✔️ Feature toggles


✅ Bitmask Usage Example
perm := Read | Write

if perm&Read != 0 {
    fmt.Println("Read allowed")
}


🔄 Reset Behavior
const (
    A = iota // 0
)

const (
    B = iota // 0 (reset)
)


🔄 Common Mistakes
❌ Forgetting iota resets per block

🔷 Embedding (Composition) and Promotion
#########################################
Embedding means including one struct inside another without naming it explicitly

Promotion means accessing embedded fields/methods directly from outer struct

🔁 Without Promotion (conceptually)
c.Engine.Power

🔁 With Promotion (actual usage)
c.Power
👉 Go automatically promotes fields/methods


❓ Why Go Uses Composition Instead of Inheritance
❌ Problems with Inheritance (Java/C++)
➤ Tight coupling
➤ Complex hierarchies
➤ Diamond problem

✅ Go Approach
➤ Simple composition
➤ Flexible design
➤ No hidden behavior

func (e Engine) Start() {
    fmt.Println("Engine started")
}

type Car struct {
    Engine
}
Usage
c := Car{}
c.Start() // method promoted
👉 Even though Start() belongs to Engine, Car can call it


🔁 Internal Working
When you write: c.Start()
👉 Compiler interprets as: c.Engine.Start()


🔄 Embedding vs Normal Field
✔️ Normal Field
type Car struct {
    engine Engine
}
👉 Usage: c.engine.Power


✔️ Embedded Field
type Car struct {
    Engine
}
👉 Usage: c.Power // promoted

    Value int
}

    Value int
}

    A
    B
}

✅ Solution

👉 Car now has:

| Type    | Behavior         |
| ------- | ---------------- |

🔷 Understanding Interfaces in Go
###################################
➤ An interface is a collection of method signatures
➤ It defines WHAT to do, not HOW to do it

🔗 Syntax
type InterfaceName interface {
    Method1()
    Method2()
}

✅ Example
type Animal interface {
    Speak() string
}


🔄 Implicit Implementation
👉 Go does NOT use implements keyword
A type automatically implements an interface if it has required methods

✅ Example
type Dog struct{}

func (d Dog) Speak() string {
    return "Woof"
}
👉 Dog automatically implements Animal

🔗 Usage
var a Animal
a = Dog{}
fmt.Println(a.Speak()) // Woof


❓ Why Interfaces are Powerful
✅ Decoupling: No dependency on concrete types
✅ Flexibility: Easily swap implementations
✅ Testability: Mock implementations for testing


🔄 Polymorphism in Go
type Animal interface {
    Speak() string
}

type Dog struct{}
type Cat struct{}

func (Dog) Speak() string { return "Woof" }
func (Cat) Speak() string { return "Meow" }

🔗 Usage
animals := []Animal{Dog{}, Cat{}}

for _, a := range animals {
    fmt.Println(a.Speak())
}
👉 Same interface → different behavior


🔄 Interface Internals
An interface is internally:

type iface struct {
    type  *typeInfo
    value unsafe.Pointer
}
👉 It stores:
    ✔️ Type
    ✔️ Value


🔄 Key Insight
var a Animal = Dog{}

👉 Interface contains:
    ✔️ Type → Dog
    ✔️ Value → {}


🔁 Nil Interface Trap
var a Animal
fmt.Println(a == nil) // true

❌ Confusing Case
var d *Dog = nil
var a Animal = d
fmt.Println(a == nil) // false ❗


❓ Why?
Type = *Dog
Value = nil
👉 Interface is NOT nil


🔄 Empty Interface (interface{})

👉 Can hold any type

x = 10

x = true

➤ It is used to assert type for an empty interface

value := x.(Type)

✅ Example

fmt.Println(str)

⚠️ Safe Assertion
str, ok := x.(string)

if ok {
    fmt.Println(str)
}


🔁 Type Switch
➤ It can be used to check against multiple types
➤ Type Assertion is done in the switch statement instead of switching on the value.
➤ Inside each case the value will of that type if case matchesm otherwise remains empty interface type.

🔗 Syntax
switch v := x.(type) {
case int:
    fmt.Println("int", v)
case string:
    fmt.Println("string", v)
default:
    fmt.Println("unknown")
}

    Write()

}

👉 Easily switch DBs without changing logic


🔄 Best Practices
✅ Keep interfaces SMALL

    Read()
}
👉 Not:
type BigInterface interface {
    Read()
    Write()
    Delete()
    Update()
}
✅ Define interfaces where used (not where implemented)
✅ Use interfaces for abstraction, not for everything

❌ Overusing empty interface
👉 Leads to loss of type safety

❌ Large interfaces
👉 Hard to maintain

❌ Ignoring nil interface behavior
👉 Causes bugs

─────────▶  ❌ No (implicit)

─────────▶  Holds any type

🔷 Mastering Error Handling in Go
#####################################
➤ Error handling in Go is explicit, predictable, and central to code quality.
➤ There are no exceptions like in Java/Python—Go treats errors as values.
➤ Errors are values → handle them explicitly → build reliable systems


🔄 Core Philosophy
➤ No try/catch
➤ Errors are returned, not thrown
➤ You must handle errors explicitly

🔗 Basic Pattern
result, err := someFunction()
if err != nil {
    // handle error
}
👉 This pattern appears everywhere in Go


🔄 The error Type
✅ Built-in Interface
type error interface {
    Error() string
}
👉 Any type implementing Error() is an error


✅ Basic Example
import "errors"

func divide(a, b int) (int, error) {
    if b == 0 {
        return 0, errors.New("division by zero")
    }
    return a / b, nil
}


🔄 Creating Errors
1️⃣ Using errors.New
err := errors.New("something went wrong")

2️⃣ Using fmt.Errorf
err := fmt.Errorf("user %d not found", id)


🔄 Custom Error Types
✅ Example
type NotFoundError struct {
    Resource string
}

func (e NotFoundError) Error() string {
    return e.Resource + " not found"
}

🔗 Usage
return NotFoundError{Resource: "User"}


❓ Why Custom Errors?
➤ Add context
➤ Structured error handling
➤ Better debugging


🔄 Error Wrapping (Go 1.13+)
➤ Wrap errosrs to add additional information.
➤ An error e wraps another error if e's type has one of the methods:
1️⃣ Unwrap() error
2️⃣ Unwrap() []error
➤ An error chain refers to series of errors that are wrapped together.

🔗 Using %w
➤ Can use fmt.Errorf with %w to wrap errors.
➤ If you don't want to wrap error but just want to add error from, use the %v placeholder in fmt.Errorf().
err := fmt.Errorf("failed to fetch user: %w", originalErr)


🔄 Using errors.Join
➤ Go 1.20 introduced errors.Join() function to wrap multiple errors.
➤ Join wraps provided error, discarding nil errors.
➤ If all errors are nil, it returns nil.
➤ A non-nil error returned by Join implements the Unwrap() []error method.
errors.Join(err1, err2)


🔄 Unwrapping
➤ Standard lib errors package has Unwrap function to unwrap errors.
➤ Unwrap calls the Unwrap method implemented on the error or returns nil.
➤ fmt.Errorof() create error implements the Unwrap method.
➤ Please note Unwrap does not unwrap errors returned by errors.Join().
errors.Unwrap(err)


🔄 Unwrapping Errors
➤ If e.Unwrap() returns a non-nil error w or a slice containing w, then we say that e wraps w.
➤ It is invalid for an Unwrap method to return an []error containing a nil error value.
➤ A nil error returned from e.Unwrap() indicates that e does not wrap any error.


🔄 Checking Errors
1️⃣ errors.Is(err, targetErr)
➤ This function accepts two parameters, the first is the error and the second the instance of error we are checking against.
➤ Function signature: func Is(err, target error) bool
➤ Iterate through the errors in the error chain and returns true  if matching error instance found.

2️⃣ errors.As(err, &targetType)
➤ This function accepts two parameters the first is the error and the second a non-nil pointer to either a type that implements error, or to any interface type.
➤ Function signature: func As(err error, target interface{}) bool
➤ Iterate through the errors in the error chain and matching target type, if found returns true and set the target to that error value.

✅ Example
if errors.Is(err, ErrNotFound) {
    fmt.Println("Handle not found case")
}


🔄 Sentinel Errors
➤ Indicate occurance of a unique events.
➤ Exported package level variable.

var ErrNotFound = errors.New("not found")

🔗 Usage

    // handle
}

1️⃣ Error (Normal Flow)
➤ Expected failure
➤ Should be handled

#####################
🔷 Panic and Recover
#####################
This is Go’s controlled failure mechanism—used rarely but critically.

➤ panic → stops normal execution
➤ recover → catches panic and resumes control (only inside defer)

Panic is a built-in function that stops the normal flow of execution and begins stack unwinding.

✅ Example
func main() {
    panic("something went wrong")
}


❓ What Happens Internally?
➤ Current function stops
➤ Deferred functions execute
➤ Stack unwinds (function by function)
➤ Program crashes if not recovered


🔄 Stack Unwinding
func A() {
    B()
}

func B() {
    panic("error in B")
}

func main() {
    A()
}

🔄 Execution Flow:
1. B() panics
2. B() exits → run its deferred calls
3. Control returns to A() → run deferred calls
4. Goes to main() → crash (if no recover)

Recover is used to catch panics and regain control.

⚠️ Important Rule
recover works ONLY inside a deferred function

🔄 Example
func main() {
    defer func() {
        if r := recover(); r != nil {
            fmt.Println("Recovered:", r)
        }
    }()

    panic("something went wrong")
}

👉 Output: Recovered: something went wrong


🔄 Panic + Recover Flow
func safe() {
    defer func() {
        if r := recover(); r != nil {
            fmt.Println("Recovered:", r)
        }
    }()

    fmt.Println("Start")
    panic("crash")
    fmt.Println("End") // never runs
}

👉 Output:
Start
Recovered: crash


🔄 Key Rules
1️⃣ Recover only works in defer
recover() // ❌ useless outside defer

2️⃣ Panic stops execution immediately

3️⃣ Deferred functions always run


🔄 Panic 🆚 Error
| Feature  | Error                      | Panic               |
| -------- | -------------------------- | ------------------- |
| Use case | Expected failures          | Unexpected failures |
| Handling | Explicit (`if err != nil`) | Automatic unwinding |
| Control  | Caller handles             | Runtime handles     |
| Usage    | Common                     | Rare                |


🔄 When to Use Panic

✅ Valid Cases
➤ Programmer errors
➤ Invalid assumptions
➤ Critical system failure

❌ Avoid Panic For
➤ User input errors
➤ API failures
➤ Database/network errors
👉 Use error instead


✅ Backend Example
func recoverMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        defer func() {
            if err := recover(); err != nil {
                log.Println("Recovered from panic:", err)
                http.Error(w, "Internal Server Error", 500)
            }
        }()

        next.ServeHTTP(w, r)
    })
}
👉 Prevents server crash



🔄 Panic in Goroutines
go func() {
    panic("goroutine crash")
}()
👉 This will crash the whole program if not recovered


✅ Safer Version
go func() {
    defer func() {
        if r := recover(); r != nil {
            fmt.Println("Recovered in goroutine:", r)
        }
    }()

    panic("goroutine crash")
}()


🔄 Internal Insight
➤ panic stores value (any type)
➤ Runtime tracks panic stack
➤ recover extracts panic value


🔄 Multiple Defers Order
func main() {
    defer fmt.Println("1")
    defer fmt.Println("2")
    panic("error")
}
✅ Output: 
2
1
panic: error
👉 LIFO(Last In First Out)


🔄 Common Mistakes
❌ Using panic for normal flow
👉 Bad practice

❌ Calling recover outside defer
👉 Won’t work

❌ Ignoring panic in goroutines
👉 Crashes app

❌ Overusing panic
👉 Makes code unpredictable

#####################################
🔷 Go Modules: Dependency Management
#####################################
➤ Go Modules are the official dependency management system in Go (introduced in Go 1.11, default after 1.13).
➤ They replace the old GOPATH-based workflow.

➤ A Go Module is a collection of Go packages stored in a directory tree with a go.mod file at its root.

➤ A module defines:
    ✔️ Project identity (module path)
    ✔️ Dependencies
    ✔️ Versions

✅ Example go.mod
module github.com/ankur/myapp

go 1.22

require (
    github.com/gin-gonic/gin v1.9.0
    github.com/go-sql-driver/mysql v1.7.1
)


❓ Why Go Modules Were Introduced
❌ Problems with GOPATH
➤ No version control
➤ Global dependency conflicts
➤ Hard to reproduce builds

✅ Advantages of Modules
➤ Versioned dependencies
➤ Reproducible builds
➤ No need for GOPATH
➤ Works anywhere (outside GOPATH)


🔄 Core Files
1️⃣ go.mod
Contains:
    ✔️ Module name
    ✔️ Go version
    ✔️ Dependencies

2️⃣ go.sum
    ✔️ Stores checksums of dependencies
    ✔️ Ensures integrity & security
    ✔️ Similar to npm's package-lock.json
    ✔️ It stores cryptographic hashes for specific modules versions.

The go.mod and go.sum files together ensure that same moduel versions are downloaded every time.


🔄 Basic Workflow
Step 1️⃣: Initialize Module

👉 Creates go.mod


Step 2️⃣: Add Dependency

Naming Convention: github.com/username/reponame
👉 Updates:
    ✔️ go.mod
    ✔️ go.sum


Step 3️⃣: Run Program
go run main.go
👉 Automatically downloads dependencies


Step 4️⃣: Clean Dependencies
go mod tidy
👉 Removes unused deps + adds missing ones

vMAJOR.MINOR.PATCH
| Part  | Meaning          |
| ----- | ---------------- |
| MAJOR | Breaking changes |
| MINOR | New features     |

#####################################
🔷 io.Reader and io.Writer in Go
#####################################
👉 Go treats everything as a stream of bytes
➤ File → stream of bytes
➤ Network → stream of bytes
➤ HTTP request → stream of bytes

➡️ io.Reader = source of data
➡️ io.Writer = destination of data


🔄 io.Reader Interface
type Reader interface {
    Read(p []byte) (n int, err error)
}

🔥 How It Works
➤ You give it a byte slice buffer (p)
➤ It fills it with data
➤ Returns:
    ✔️ n → number of bytes read
    ✔️ err → error (or io.EOF when finished)

✅ Example: Reading from a string
package main

import (
    "fmt"
    "strings"
)

func main() {
    r := strings.NewReader("Hello Go")

    buf := make([]byte, 4)

    for {
        n, err := r.Read(buf)
        if err != nil {
            break
        }
        fmt.Println(string(buf[:n]))
    }
}
✅ Output
Hell
o Go


🔄 Common Implementations
1️⃣ *os.File: used to read data from a file.
2️⃣ strings.Reader: used to read data from a string.
3️⃣ bufio.Reader: implements Read for imporved performance, using buffers for reading.
4️⃣ http.Request.Body: allows you to supply the request body via reader.
5️⃣ bytes.Buffer: used for reading from in-memory byte buffers.


⚠️ Important Behavior
➤ Read() may return less data than buffer size
➤ Must always use: buf[:n]


🔄 io.Writer Interface
type Writer interface {
    Write(p []byte) (n int, err error)
}

🔥 How It Works
➤ You pass data ([]byte)
➤ Writer writes it somewhere
➤ Returns:
    ✔️ n → bytes written
    ✔️ err → error if failed
➤ If written bytes are less than the provided bytes, you should return an error.
➤ Provided slice of byte is a source of writing and should not be modified.

✅ Example: Writing to buffer
package main

import (
    "bytes"
    "fmt"
)

func main() {
    var b bytes.Buffer

    b.Write([]byte("Hello "))
    b.Write([]byte("Go"))

    fmt.Println(b.String())
}
✅ Output
Hello Go

| HTTP Body          | Reader          |
| Network Connection | Reader + Writer |

    // works with file, network, string, etc.
}

🔷 JSON Marshal, Unmarshal, and Encoding/Decoding
##################################################
👉 Go uses the encoding/json package
➤ Marshal → Go struct ➝ JSON
➤ Unmarshal → JSON ➝ Go struct
➤ Encoder/Decoder → Stream-based JSON processing (I/O based)


🔄 JSON Marshal (Struct ➝ JSON)
🔗 Function
func Marshal(v interface{}) ([]byte, error)

✅ Example
package main

import (
    "encoding/json"
    "fmt"
)

type User struct {
    Name string
    Age  int
}

func main() {
    u := User{Name: "Ankur", Age: 22}

    data, _ := json.Marshal(u)
    fmt.Println(string(data))
}
✅ Output: {"Name":"Ankur","Age":22}

⚠️ Important Rules: Only exported fields are marshaled (Name ✅, name ❌)


🔄 Struct Tags
👉 Control JSON output

type User struct {
    Name string `json:"name"`
    Age  int    `json:"age,omitempty"`
}


🔁 Tag Features
➤ json:"name" → rename field
➤ omitempty → skip if zero value
➤ - → ignore field

✅ Example
type User struct {
    Name string `json:"name"`
    Age  int    `json:"-"`
}


🔄 JSON Unmarshal (JSON ➝ Struct)
🔗 Function
func Unmarshal(data []byte, v interface{}) error

✅ Example
package main

import (
    "encoding/json"
    "fmt"
)

type User struct {
    Name string
    Age  int
}

func main() {
    jsonData := []byte(`{"Name":"Ankur","Age":22}`)

    var u User
    json.Unmarshal(jsonData, &u)

    fmt.Println(u.Name, u.Age)
}

⚠️ VERY IMPORTANT
Must pass pointer: json.Unmarshal(data, &u) // ✅

name := data["name"].(string)

🔄 Encoder & Decoder
❓ Why Use Them?
➤ Handle large data
➤ Work with io.Reader / io.Writer
➤ Avoid loading full JSON in memory


1️⃣ Encoder (Write JSON)
json.NewEncoder(w io.Writer).Encode(v)
✅ Example

2️⃣ Decoder (Read JSON)
json.NewDecoder(r io.Reader).Decode(&v)
✅ Example

🔥 Key Difference
| Method  | Use Case               |
| ------- | ---------------------- |
| Marshal | Small data             |
| Encoder | Large / streaming data |

👉 Implement Interfaces

}

3️⃣ Use struct whenever possible
4️⃣ Reuse buffers if needed

🔷 Working with Time in Go
###########################
Time handling in Go is built around the time package, and it's extremely powerful + widely used in backend systems (APIs, logs, scheduling, DB timestamps).

🔄 Core Type: time.Time
👉 Represents a specific point in time
t := time.Now()

🔁 Key Properties
➤ Stores:
    ✔️ Date
    ✔️ Time
    ✔️ Timezone
➤ Immutable (returns new values on modification)


🔄 Getting Current Time
t := time.Now()
fmt.Println(t)
✅ Output Example
2026-04-20 21:30:00.123456 +0530 IST


🔄 Creating Custom Time
t := time.Date(2026, time.April, 20, 10, 30, 0, 0, time.UTC)
✅ Parameters Order
Year, Month, Day, Hour, Min, Sec, Nanosec, Location


🔄 Time Formatting
👉 Go uses a reference time: Mon Jan 2 15:04:05 MST 2006
✅ Format Example
t := time.Now()
fmt.Println(t.Format("2006-01-02 15:04:05"))


🔄 Common Formats
"2006-01-02" → Date
"15:04:05" → Time
"02 Jan 2006" → Custom


🔄 Trap
➤ Not YYYY-MM-DD
➤ Must use reference numbers


🔄 Parsing Time (String ➝ Time)
layout := "2006-01-02"
t, _ := time.Parse(layout, "2026-04-20")
⚡ Important: Layout must match input format exactly


🔄 Time Arithmetic
1️⃣ Add Time

diff := t2.Sub(t)

2h0m0s
