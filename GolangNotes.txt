рҹ§  Go Design Philosophy

рҹҸ— Go Architecture (VERY IMPORTANT)
рҹ”Ғ High-Level Go Architecture
Go Source Code (.go)
        вҶ“
      Compiler
        вҶ“
     Machine Code
        вҶ“

рҹ”Ғ 1. Go Compiler
    вң”пёҸ Ahead-of-Time (AOT) compiled
    вң”пёҸ No JVM / no bytecode
    вң”пёҸ Produces single native binary

Advantages
    вң”пёҸ Fast startup рҹҡҖ
    вң”пёҸ No runtime dependency
    вң”пёҸ Easy deployment

вң… Example:

./main


рҹ”Ғ 2. Go Runtime
Unlike Java, Go runtime is small and focused.
Go Runtime Manages:
    вң”пёҸ Goroutines scheduling
    вң”пёҸ Garbage Collection
    вң”пёҸ Memory allocation
    вң”пёҸ Channel communication
    вң”пёҸ Stack management
вҡ пёҸ You donвҖҷt manage runtime directly вҖ” Go does.


рҹ”Ғ 3. Go Scheduler (GMP Model)
The Famous G-M-P Model
| Component | Meaning                 |
| --------- | ----------------------- |
| G         | Goroutine               |
| M         | OS Thread               |
| P         | Processor (logical CPU) |
G (goroutine)
   вҶ“
P (scheduler)
   вҶ“
M (OS thread)


вқ“ Why this matters?
вһӨ Millions of goroutines can run on few OS threads
вһӨ Extremely lightweight concurrency
вһӨ No thread explosion


рҹ”Ғ 4. Memory Architecture
Stack vs Heap in Go
вһӨ Goroutines start with small stack (~2KB)
вһӨ Stack grows/shrinks dynamically
вһӨ Escape analysis decides:
    вң”пёҸ Stack allocation вң…
    вң”пёҸ Heap allocation вқҢ
рҹ‘ү Improves performance significantly.

рҹ”Ғ package main
A package is:
    вң”пёҸ A unit of code organization
    вң”пёҸ A namespace
    вң”пёҸ A compilation boundary
Every .go file must belong to exactly one package.


вқ“ Why package main?
вһӨ main is special in Go.
вһӨ If a program has:
    вң”пёҸ package main
    вң”пёҸ func main()
вһЎпёҸ Go treats it as an executable program, not a library.
вқҢ Without package main, Go produces a library, not a runnable binary.
рҹ‘ү Rule:Only one main package per executable.

4пёҸвғЈ Exported vs Unexported Identifiers

рҹ”· Unveiling Slices in Go
##########################
вһӨ A dynamic view over an array
вһӨ A descriptor, not a data container itself
вһӨ The most commonly used collection type in Go

вң… Declaration
var arr = []int{1, 2, 3}


рҹ”„ Slice vs Array
| Feature              | Array | Slice             |
| -------------------- | ----- | ----------------- |
| Size                 | Fixed | Dynamic           |
| Type includes length | вң”     | вқҢ               |
| Passed by value      | вң”     | Descriptor copied |
| Real-world usage     | Rare  | Very common       |


рҹ”„ Slice Internal Structure
A slice internally has three components:
type slice struct {
    pointer  *T   // points to array
    length   int
    capacity int
}
вҡ пёҸ Key Insight: Passing a slice copies this struct, NOT the underlying array.


рҹ”„ Visual Representation
Underlying Array: [10 20 30 40 50]
Slice:             вҶ‘
pointer = index 1
len = 2
cap = 4


рҹ”„ Declaring & Initializing Slices
1пёҸвғЈ Nil Slice
var s []int

вһӨ len(s) == 0
вһӨ cap(s) == 0
вһӨ s == nil в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ true


2пёҸвғЈ Empty Slice
s := []int{}

вһӨ len == 0
вһӨ cap == 0
вһӨ s != nil


3пёҸвғЈ Using make
s := make([]int, 5)

вһӨ Length = 5
вһӨ Capacity = 5

s := make([]int, 3, 10)

вһӨ Length = 3
вһӨ Capacity = 10


4пёҸвғЈ Slice Literal
s := []int{1, 2, 3}


рҹ”„ Slicing an Array or Slice
arr := [5]int{10, 20, 30, 40, 50}
s := arr[1:4]

Result: [20 30 40]


1пёҸвғЈ Slice Expression Rules
s := arr[low : high]

вһӨ Includes low
вһӨ Excludes high


Full Slice Expression (Advanced)
s := arr[1:3:4]

вһӨ len = high - low
вһӨ cap = max - low


рҹ”„ Length vs Capacity
s := []int{1,2,3,4}
fmt.Println(len(s)) // 4
fmt.Println(cap(s)) // 4

Capacity Growth (append)
s := []int{}
s = append(s, 1)

вһӨ Capacity grows automatically
вһӨ Growth strategy is implementation-dependent


рҹ”„ append() вҖ“ How It Really Works
Case 1пёҸвғЈ: Capacity Available
вһӨ Element added
вһӨ Same underlying array


Case 2пёҸвғЈ: Capacity Full
вһӨ New array allocated
вһӨ Old data copied
вһӨ Slice now points to new array


вң… Trap Example
a := []int{1,2,3}
b := append(a, 4)

a[0] = 100
fmt.Println(b) // ?
вҡ пёҸ Depends on capacity!


рҹ”„ Slices are Reference-Like
func modify(s []int) {
    s[0] = 100
}

func main() {
    a := []int{1,2,3}
    modify(a)
    fmt.Println(a) // [100 2 3]
}
вқ“ Why?
вһӨ Slice descriptor copied
вһӨ Underlying array shared


рҹ”„ Copying Slices Correctly
src := []int{1,2,3}
dst := make([]int, len(src))
copy(dst, src)

вқҢ Wrong Copy

##############################
рҹ”· вҖңRipple EffectвҖқ in Golang
##############################
It means that: Changes made to one slice may unintentionally affect another slice, because both slices share the same underlying array.

This happens due to Go's slice architecutre.


рҹ”„ Before Ripple Effect в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ Understand Slice Internals (CRITICAL)
вһӨ A slice is NOT an array.
вһӨ Internally, a slice is a descriptor (header) with three fields:

type slice struct {
    pointer  *array  // points to underlying array
    length   int
    capacity int
}
So when you write: s := []int{10, 20, 30, 40}

Memory looks like: Underlying Array: [10, 20, 30, 40]
                   вҶ‘
                   s (len=4, cap=4)


рҹ”„ Creating a Slice From Another Slice
s1 := []int{10, 20, 30, 40}
s2 := s1[1:3]

Slice Header Values
| Slice | Points To | Len | Cap |
| ----- | --------- | --- | --- |
| s1    | index 0   | 4   | 4   |
| s2    | index 1   | 2   | 3   |

Memory:
Underlying Array: [10, 20, 30, 40]
                    вҶ‘        вҶ‘
                   s2       s2 end
рҹ‘ү Same array, different view


вң… Ripple Effect Example
package main

import "fmt"

func main() {
    s1 := []int{10, 20, 30, 40}
    s2 := s1[1:3]

    s2[0] = 999

    fmt.Println("s1:", s1)
    fmt.Println("s2:", s2)
}
вң… Output
s1: [10 999 30 40]
s2: [999 30]


вқ“ Why did s1 change?
вһӨ s2[0] points to s1[1]
вһӨ Same underlying array
вһӨ Modification ripples back
вһЎпёҸ This is the Ripple Effect


рҹ”„ Visual Explanation
s1 = [10, 20, 30, 40]
          вҶ‘
        s2[0]
рҹ‘ү Changing s2[0] modifies index 1 of the array в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ reflected in s1.


рҹ”„ Ripple Effect with append()
Case 1пёҸвғЈ: Capacity Available в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ Ripple Happens
s1 := []int{1, 2, 3, 4}
s2 := s1[1:3] // len=2, cap=3

s2 = append(s2, 999)

fmt.Println(s1)
fmt.Println(s2)

вң… Output
[1 2 3 999]
[2 3 999]

вқ“ Why?
вһӨ s2 had extra capacity
вһӨ append reused same underlying array
вһӨ 999 placed at index 3
вһӨ s1 affected


Case 2пёҸвғЈ: Capacity Exhausted в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ NO Ripple
s1 := []int{1, 2, 3, 4}
s2 := s1[1:3:3] // Full slice expression в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ cap limited

s2 = append(s2, 999)

fmt.Println(s1)
fmt.Println(s2)

вң… Output
[1 2 3 4]
[2 3 999]

вқ“ Why?
вһӨ Capacity restricted
вһӨ New array allocated
вһӨ No shared memory
вһӨ No ripple


рҹ”„ Full Slice Expression
s2 := s1[low : high : max]
вһӨ len = high - low
вһӨ cap = max - low
вһӨ Prevents accidental overwrite


рҹ”„ Avoiding Ripple Effect
Method 1пёҸвғЈ: Use copy()
original := []int{10, 20, 30}
safe := make([]int, len(original))
copy(safe, original)

Now:
    вң”пёҸ Separate arrays
    вң”пёҸ No ripple


Method 2пёҸвғЈ: Force new array with append
safe := append([]int{}, original...)

вқҢ Dangerous Pattern (COMMON BUG)
a := []int{1, 2, 3}
b := a[:2]
b[0] = 100
рҹ‘ү Developers expect a unchanged вҖ” WRONG

вқ“ Why Go Allows This
Go prioritizes:
    вң”пёҸ Performance
    вң”пёҸ Low allocations
    вң”пёҸ Predictable memory usage

Sharing arrays:
    вң”пёҸ Avoids copying
    вң”пёҸ Faster slicing
    вң”пёҸ Responsibility on developer
рҹ‘ү Go gives you power, not safety by default


вқ“ Questions
1пёҸвғЈ Why does modifying one slice affect another?
вң”пёҸ Because slices share the same underlying array.


2пёҸвғЈ When does append allocate a new array?
вң”пёҸ When capacity is insufficient.


3пёҸвғЈ How to avoid ripple effect?
вң”пёҸ Use copy() or restrict capacity using full slice expression.


4пёҸвғЈ Is slice pass-by-value or reference?
вң”пёҸ Slice header is passed by value, but it points to shared memory.

рҹ”· Connections between Strings, Runes and Bytes
################################################
In Go, strings, bytes, and runes are tightly connected, but they are NOT the same thing.
| Concept  | Represents                  | Size              |
| -------- | --------------------------- | ----------------- |
| `string` | Read-only sequence of bytes | variable          |
| `byte`   | Raw binary data             | 1 byte (`uint8`)  |
| `rune`   | A Unicode code point        | 4 bytes (`int32`) |
Go strings are UTF-8 encoded byte sequences


рҹ”„ String in Go?
s := "Hello рҹҢҚ"

Internally: A string is:
type string struct {
    ptr *byte
    len int
}
вһӨ Immutable
вһӨ Stored as UTF-8 encoded bytes


рҹ”— Length of a String
fmt.Println(len("Hello"))     // 5
fmt.Println(len("рҹҢҚ"))        // 4 вқ—

вқ“ Why?
вһӨ рҹҢҚ is 1 character
вһӨ But 4 bytes in UTF-8


рҹ”„ Byte: The Lowest Level
b := []byte("Hello рҹҢҚ")
fmt.Println(b)

Output: [72 101 108 108 111 32 240 159 140 141]

рҹ‘ү Key Points
вһӨ Each number is a byte
вһӨ UTF-8 characters may take 1вҖ“4 bytes
вһӨ byte = alias for uint8


Iterating Over Bytes (вқҢ for Unicode)
for i := 0; i < len(s); i++ {
    fmt.Printf("%c ", s[i])
}
Output: H e l l o   пҝҪ пҝҪ пҝҪ пҝҪ
рҹҡЁ Broken for Unicode


рҹ”„ Rune: Unicode Code Point
r := 'рҹҢҚ'
fmt.Println(r)        // 127757
fmt.Printf("%T", r)   // int32

рҹ‘ү Key Points
вһӨ rune = alias for int32
вһӨ Represents one Unicode character
вһӨ Can hold any Unicode symbol

Iterating Over Runes (вң… Correct)
for _, ch := range "Hello рҹҢҚ" {
    fmt.Printf("%c ", ch)
}
Output: H e l l o   рҹҢҚ


рҹ”„ range вҖ” The Connector Between Them
When you use range on a string:
for i, r := range s {
    fmt.Println(i, r)
}

вқ“ What happens internally?
вһӨ Decodes UTF-8 bytes
вһӨ Converts to rune
вһӨ Moves index by rune byte length
вһЎпёҸ range = UTF-8 aware


рҹ”„ String в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ Rune Slice в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ String
s := "GoрҹҢҚ"
r := []rune(s)
fmt.Println(len(s))   // 6 bytes
fmt.Println(len(r))   // 3 runes

вқ“ Why convert?
вһӨ Safe character indexing
вһӨ Replacing characters

вң… Example: Modify a String (Strings are immutable)
s := "Hello рҹҢҚ"
r := []rune(s)
r[6] = 'рҹҡҖ'
s = string(r)
fmt.Println(s)

вң… Output: Hello рҹҡҖ


рҹ”„ String Indexing Pitfall
s := "рҹҢҚ"
fmt.Println(s[0])

вң… Output: 240

вқ“ Why?
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ s[0] gives first byte,Not the rune

вң… Correct Way
r := []rune(s)
fmt.Println(r[0])     // 127757
fmt.Printf("%c", r[0])


рҹ”„ Visual Memory Model
"рҹҢҚ" (string)
вҶ“ UTF-8
[240][159][140][141]  вҶҗ bytes
вҶ“ decoding
127757                вҶҗ rune


рҹ”„ When to Use What
Use string when:
вһӨ Read-only text
вһӨ Logging
вһӨ Keys, messages, JSON


Use []byte when:
вһӨ File I/O
вһӨ Network data

Use []rune when:

рҹ”„ Questions
1пёҸвғЈ Why len("рҹҢҚ") == 4?
вң”пёҸ Because strings store UTF-8 bytes.


2пёҸвғЈ What does range return when iterating over string?

рҹ”· Demystifying Maps in Golang
################################
Maps are one of the most used data structures in Go вҖ” especially in backend, APIs, caching, configuration, and data processing.

If you master maps, you master:
    вң”пёҸ Hashing behavior
    вң”пёҸ Reference semantics
    вң”пёҸ Concurrency pitfalls
    вң”пёҸ Performance optimization


рҹ”„ What is a Map in Go?
A map is a built-in data structure that associates keys with values. It is implemented as a hash table and provides efficient lookups, insertions, and deletions.
key в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ value

вң… Example:
m := map[string]int{
    "apple":  10,
    "banana": 20,
}

рҹ”„ Declaring and Initializing Maps
1пёҸвғЈ Zero Value Map
var m map[string]int

вһӨ m == nil в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ true
вһӨ Cannot insert values into nil map в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ panic
вһӨ Reading from nil map returns zero value в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ 0
вһӨ Writing to nil map causes runtime panic
m["key"] = 100 // вқҢ panic: assignment to entry in nil map


рҹ”„ Initialization of Empty Map
1пёҸвғЈ Using Literal
m := map[string]int{}
вһӨ Now it is initialized and can be used safely.

2пёҸвғЈ Using make()
m := make(map[string]int)
m["apple"] = 10
вһӨ Now it is initialized and can be used safely.

3пёҸвғЈvar m map[string]int= map[string]int{}


рҹ”„ With Initial Capacity
m := make(map[string]int, 100)
вһӨ Pre-allocates space for 100 entries
вһӨ Performance boost if size known
вһӨ Capacity is a hint, not a fixed size


4пёҸвғЈ Map Literal
m := map[string]int{
    "apple":  10,
    "banana": 20,
}


рҹ”„ Map Internal Architecture
Go maps are implemented as hash tables with separate chaining for collision resolution.
Internally: key в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ hash function в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ bucket в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ value

рҹ”Ғ Core Concepts
вһӨ Uses hashing
вһӨ Data stored in buckets
вһӨ Each bucket stores multiple key-value pairs
вһӨ Buckets linked when overflow happens
вһӨ Automatically resizes (rehashes)


рҹ”„ Key Rules for Map Keys
1пёҸвғЈ Must be Comparable types
вң… Allowed: bool, int, float, string, pointers, structs (if all fields comparable)
вқҢ Not allowed: slices, maps, functions


рҹ”„ Reading From Map
v:= m["key"]
вһӨ If key exists в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ returns value
вһӨ If key does not exist в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ returns zero value of value type (e.g., 0 for int, "" for string)


рҹ”„ Comma-OK Idiom
v, ok := m["key"]
вһӨ ok == true в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ key exists
вһӨ ok == false в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ key does not exist
This is crucial to distinguish between a missing key and a key with zero value.

вң… Example:
m := map[string]int{
    "apple": 10,
}
v, ok := m["banana"]
fmt.Println(v)  // 0 (zero value)
fmt.Println(ok) // false (key does not exist)


рҹ”„ Writing to Map
m["key"] = value
вһӨ If key exists в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ updates value
вһӨ If key does not exist в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ creates new entry


рҹ”„ Deleting from Map
delete(m, "key")
вһӨ Removes key-value pair if key exists
вһӨ No error if key does not exist


рҹ”„ Iterating Over Map
for k, v := range m {
    fmt.Println(k, v)
}
вһӨ Iteration order is random в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ do NOT rely on it
вһӨ Each run may produce different order


рҹ”„ Map Are Reference Types
When you assign a map to another variable, both variables point to the same underlying data structure.
m1 := map[string]int{
    "apple": 10,
}
m2 := m1
m2["banana"] = 20
fmt.Println(m1) // map[apple:10 banana:20]
fmt.Println(m2) // map[apple:10 banana:20]

Map behaves like: slices, pointers, channels, functions (reference semantics).


рҹ”„ Passing Maps to Functions
func modify(m map[string]int) {
    m["x"] = 500
}
Change is visible outside the function because map header is passed by value but underlying structure is a reference type.


рҹ”„ Comparing Maps
Maps cannot be compared directly using == operator (except for nil).
вқҢ m1 == m2 // compile error
вң”пёҸ Only comparison allowed is with nil: m == nil

вң… To compare maps, you must write a custom function that checks keys and values.


рҹ”„ Concurrency Warning
Maps are NOT safe for concurrent use by default. If multiple goroutines access a map simultaneously and at least one of them modifies the map, it can lead to a race condition and cause a runtime panic.
fatal error: concurrent map writes

If multiple goroutines access map:
    вң”пёҸ Use sync.Mutex
    вң”пёҸ Or sync.RWMutex
    вң”пёҸ Or sync.Map


рҹ”„ When to use sync.Map
вң… Use sync.Map when:
    вң”пёҸ High contention
    вң”пёҸ Many goroutines
    вң”пёҸ Read-heavy workloads
    вң”пёҸ When you want built-in concurrency safety without manual locking


рҹ”„ Map Resizing and Growth
When:
    вң”пёҸ Load factor increases
    вң”пёҸ Too many collisions

Go:
    вң”пёҸ Allocates new larger bucket array
    вң”пёҸ Gradually rehashes elements
    вң”пёҸ Uses incremental growth
рҹ‘ү This prevents: Big latency spikes


рҹ”„ Zero Value Behavior
вң… Example:
m := make(map[string]int)
fmt.Println(m["unknown"])

вң… Output: 0
рҹ‘ү Because: Zero value of int is 0

For bool map:
m := make(map[string]bool)
fmt.Println(m["x"]) // false


рҹ”„ Map of Structs vs Map of Pointers
map[string]User
map[string]*User

Use pointer version when:
    вң”пёҸ Struct is large
    вң”пёҸ Frequent modifications
    вң”пёҸ Performance sensitive


рҹ”„ Performance Insights
Average complexity:
    вң”пёҸ Lookup в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ O(1)
    вң”пёҸ Insert в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ O(1)
    вң”пёҸ Delete в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ O(1)
Worst case:  O(n) (heavy collisions)
